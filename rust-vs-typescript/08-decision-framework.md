# 08 · 决策框架：要不要在我们项目里上 Rust？

> 前 7 章铺垫完了。这一章给你一套**可执行的决策流程** —— 不再是"Rust 强还是 TS 强"的抽象比较，而是"我们的具体情况下，该投入多少、什么时候投入"。

---

## 1. 先承认一个事实

> **大多数业务系统，TS 已经够用。**

如果你的系统满足以下**全部条件**，**就继续 TS**，别折腾：

- ☐ 当前没有精度问题导致的对账差错
- ☐ 性能瓶颈不在 CPU 上（多数瓶颈是 DB / 网络 / 缓存）
- ☐ 团队全员 TS 熟练，没人对 Rust 感兴趣
- ☐ 业务节奏快，没空给团队 6 个月学习期
- ☐ 招人渠道里 Rust 工程师稀少

满足**4 条以上 → 继续 TS**。引入 Rust 是负 ROI。

---

## 2. 反过来：什么信号说明该考虑 Rust

如果你看到下面**任何 2 条以上**，Rust 的 ROI 开始正向：

| 信号 | 意味着 |
| --- | --- |
| 浮点精度问题在生产爆过 | TS 的"自律 + 库"防线已经漏过 |
| 批量结算/对账要跑几个小时 | CPU 瓶颈，Node 单线程无解 |
| Node 进程内存频繁 OOM | GC 压力大，Rust 内存少 10x |
| 启动时间 > 1 秒影响 Serverless | Rust 启动 < 10ms 完美 |
| 数据竞争 BUG 偶尔出现 | Rust 编译期消灭这类 |
| 团队里已经有 1-2 人在自学 Rust | 种子选手有了 |
| 业务要求扩到嵌入式 / 边缘端 | Node 几乎不可能，Rust 天然适合 |
| 同类竞品已经用 Rust 并明显赢你 | 性能优势变商业优势 |

## 3. 决策树

```
你们系统的核心瓶颈是什么？
│
├── DB / 网络 / 缓存
│    └── 别上 Rust，先优化 SQL / 加缓存 / 拆服务
│
├── 单一节点 CPU
│    │
│    ├── 是高频小请求 (实时报价、定价 API)
│    │    │
│    │    ├── 团队有 Rust 储备 → napi-rs 单点替换
│    │    └── 没有 → 培养种子人 + PoC
│    │
│    └── 是低频大批量 (日终结算、月度精算)
│         │
│         ├── 数据流是文件 (CSV / Parquet) → Rust CLI 工具
│         └── 数据来自 DB → Rust 微服务 / napi-rs
│
├── 浮点精度问题
│    │
│    ├── 痛但能凑合 → 先在 TS 引入 decimal.js + 强制 lint
│    └── 已经出过对账事故 → 核心计算迁 Rust（强制 Decimal）
│
└── 没有明显瓶颈，但想"准备未来"
     └── 别上。准备未来的最好方式是把当前业务做好。
```

```
[现状]: TS 实现，单机吞吐 ___ QPS，批量结算耗时 ___，过去 6 个月发生
        过 ___ 次精度/性能相关事故

[方案对比]:
  方案 A 优化 TS:    成本 ___ 人月, 预期收益 ___, 可达性 ___
  方案 B 单点引 Rust: 成本 ___ 人月, 预期收益 ___, 可达性 ___
  方案 C 大规模迁:    成本 ___ 人月, 预期收益 ___, 可达性 ___

[推荐]: 方案 B (理由: 风险可控、可灰度、不破坏现有架构)

[里程碑]:
  - 第 1 月: 2 人学习 + PoC 验证
  - 第 2-3 月: 单一计算函数迁移，双跑对比

  - 维护双栈 → 严格控制 Rust 模块边界，不让蔓延
```

## 5. 别犯的几个常见错

### 5.1 "重写综合症"

错的版本：**"我们整个项目用 Rust 重写！"**

为什么错：
- 业务暂停 6-12 个月
- 重写期 BUG 暴增（业务规则跟代码绑定，规则不是文档化的）
- 完成时业务已经又变了，重写的也过时
- 团队几个核心人扛不住一起离职

对的版本：**"我们把 X 函数用 Rust 写，napi 引进来，跑稳后再考虑下一个。"**

### 5.2 "技术追潮"

错的版本：**"Rust 很火，Discord 在用，AWS 在用，我们也要用。"**

为什么错：
- 别人的场景跟你不一样
- 你的痛点可能根本不是 Rust 解的
- 团队没准备好，烧人

对的版本：**"我们生产爆过 3 次精度 BUG，对账损失 X 万。我们尝试用 Rust 的 Decimal 类型彻底解决。"**

### 5.3 "二选一"

错的版本：**"要么全 TS 要么全 Rust，不能混。"**

为什么错：
- 行业最佳实践就是混（Discord 是 Python + Rust + Elixir 混）
- 不同问题用不同工具是常识
- "技术栈统一"听着好，实际是把所有问题塞进一个工具

对的版本：**"业务编排和 CRUD 用 TS；核心计算用 Rust。两栈共存，边界清晰。"**

### 5.4 "等团队都会再开始"

错的版本：**"等大家都学会 Rust 我们再开始。"**

为什么错：
- 没有真项目，没人会真学
- 永远等不到那天

对的版本：**"2 个种子人先做 PoC，跑通后带着大家做第一个生产项目，边做边学。"**

## 6. 如果你决定开始，下一步该做什么

按顺序：

### 第 1 周
- [ ] 找出团队里 1-2 个对 Rust 有兴趣 / 有储备的人
- [ ] 列出系统当前 3 个最痛的具体问题，量化（精度事故次数、批量耗时、内存峰值）
- [ ] 写一份"为什么我们要尝试 Rust"的内部文档，跟 leader / 团队对齐预期

### 第 2-4 周
- [ ] 种子人完成 [03-syntax-tour.md](03-syntax-tour.md) + [04-ownership.md](04-ownership.md) + [05-error-handling.md](05-error-handling.md) + [06-decimal-precision.md](06-decimal-precision.md)
- [ ] 挑 1 个具体函数（比如保费基础公式）做 PoC
- [ ] napi-rs 包装，本地跑通

### 第 2 个月
- [ ] PoC 双跑对比 TS 和 Rust 版本（结果一致性 + 性能差距）
- [ ] 写 PoC 复盘文档：实际收益、踩过的坑、团队反馈

### 第 3 个月
- [ ] 如果 PoC 顺利 → 接入到测试环境
- [ ] 如果团队反馈良好 → 灰度上线（小流量）
- [ ] 不顺利也没关系 → 至少团队多了一个评估视角

### 第 4-6 个月
- [ ] 灰度扩大，监控指标
- [ ] 迁第 2、3 个函数
- [ ] 沉淀团队规范（Cargo workspace 结构、错误处理风格、测试套路）

### 6 个月评估
- [ ] 真实收益：性能 / 精度 / BUG 数 / 团队感受
- [ ] 决定是否扩大投入（更多功能 / 招聘 Rust 人 / 独立微服务）

## 7. 不上 Rust 也能改善的事

如果你最终决定不上 Rust，下面这些事在 TS 里也能做，**先把这些做完再考虑**：

- [ ] 启用 `tsconfig.json` 的 `strict: true` 全部子选项
- [ ] 给所有金额引入 `decimal.js`，并写 lint 规则禁止 `number` 用于金额字段
- [ ] 关键计算函数加 property-based testing (fast-check)
- [ ] 性能瓶颈用 `worker_threads` 切多核
- [ ] CPU profiling 找 hot path 优化
- [ ] 大批量场景拆 Node 进程做并行
- [ ] 严格异常处理规范 + 强制 `try-catch` lint

很多团队"想引入 Rust" 的本质需求是"想要更严格"，**TS 把 strict 拉满 + 工程规范严格后能拿到 70% 的好处**。把这些做完再决定要不要 30% 的剩余收益。

## 8. 最终的两段话

**给"还在犹豫"的你**：

> Rust 不是宗教信仰，是一个工具。它在**特定场景**下（金额精度、CPU 密集、长生命周期、高正确性）有别的语言难以替代的优势。但代价真实存在 —— 学习成本、招聘难度、生态相对窄。先把痛点量化，再决定值不值得付这个代价。

**给"已经决定要上"的你**：

> 不要追求"一次到位"。从最小可验证的范围开始（一个函数 + napi-rs），跑稳后再扩。让团队**通过真项目**学 Rust，而不是先关在屋里学 6 个月。失败的引入往往不是技术问题，是节奏问题 —— 一开始野心太大、铺得太广、回头路太难。
