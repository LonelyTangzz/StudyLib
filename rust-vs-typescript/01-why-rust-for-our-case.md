# 01 · 为什么在保费保单计算场景里值得看 Rust

> 不是因为 Rust 火。是因为保费计算这类系统的几个固有痛点，TS 天然就难处理；而 Rust 的设计目标几乎对应每一个痛点。

---

## 1. 保费保单计算系统的"性格"

不管业务怎么变，这类系统通常具备以下特征（你对照下自己的项目）：

| 特征 | 具体表现 |
| --- | --- |
| **数据高度结构化** | 投保人、险种、保单条款、费率表、风险因子……每个都有严格 schema |
| **规则多且嵌套** | 不同险种、不同地区、不同年龄段、不同附加险，规则组合爆炸 |
| **金额必须精确** | 一分钱不能错；浮点误差到对账阶段就是灾难 |
| **批量量大** | 月底批量结算、续保提醒、再保分摊，几十万到几千万保单 |
| **结果可追溯** | 任何一个数字算错都要能复盘"是哪一步、用什么参数、走的哪条规则" |
| **变更密集** | 业务规则随合规/产品迭代变，但**变了就不能错** |

## 2. TS 在这些场景里的真实痛点

### 2.1 浮点精度

```typescript
// 看似简单的保费计算
const base = 1000.0;
const factor = 0.1;
const premium = base * factor + 0.2;  // 期望 100.2
console.log(premium);                 // 实际 100.19999999999999
```

JavaScript / TypeScript 的 `number` 是 IEEE 754 双精度浮点。它的根本问题是 `0.1 + 0.2 !== 0.3`。

**实际工程后果**：
- 单笔保费偏差 0.01 → 一百万保单批量结算偏差累计上千
- 跟核保 / 财务对账时核不平
- 写测试时不能用 `===` 比较，要 `Math.abs(a - b) < 1e-9`，新人忘了就埋雷

**TS 的解决方案**：用 `decimal.js`、`big.js`、或自己包整数（"分"为单位）。但这些方案**都是约定**，不是语言强制：
- 团队里只要一个人写了 `+ 0.5` 就破功
- 第三方库返回 `number` 你照样得手动转
- JSON 序列化反序列化是 `number ↔ Decimal` 的脆弱边界

### 2.2 错误被悄悄吞掉

```typescript
function calculatePremium(policy: Policy): number {
  const base = lookupBaseRate(policy.type);   // 可能 throw
  const ageFactor = ageMultiplier(policy.age); // 可能返回 undefined
  return base * ageFactor;                     // ⚠️ undefined * number = NaN，且不报错
}
```

- `throw` 不在签名里，调用方不知道要 try-catch
- `undefined` / `null` 渗入运算，结果是 `NaN`，**继续往下传**直到入库 / 报表才被发现
- TS 的 `strictNullChecks` 帮一部分忙，但**异常仍然是隐形的**

### 2.3 并发模型限制

Node 是单线程事件循环 + 异步 I/O：
- I/O 密集（数据库、网络）很合适
- **CPU 密集（批量精算）单核跑死**

绕路方案：
- `worker_threads` —— 真多线程但通信走消息，写起来啰嗦，对象拷贝有开销
- 拆微服务横向扩 —— 运维复杂度暴涨
- 写 native 模块 —— 那其实就是写 C++/Rust 了

### 2.4 重构期的"沉默退化"

业务规则变了，改一处函数签名：
- TS 严格模式能帮一部分
- 但只要某个调用方做了 `as any` 或 `// @ts-ignore`，破窗就开始
- **运行时退化** 只能靠测试覆盖发现，没覆盖到的悄无声息

### 2.5 性能天花板

V8 已经很强，但比原生编译语言：
- 启动慢（冷启动 500ms-1s）
- 内存占用大（几十 MB 起）
- 跑大量纯计算时被 5-50 倍甩开

对实时计费、高频报价、大批量精算，是真实瓶颈。

---

## 3. Rust 怎么回应这些痛点

| 痛点 | Rust 的回应 |
| --- | --- |
| 浮点精度 | `rust_decimal` 是类型，**编译期不能跟 f64 混算** |
| 异常被吞 | 没有 exception，所有可失败操作返回 `Result<T, E>`，**不处理编译器报错** |
| 没有 null | 用 `Option<T>`，编译期强制处理 None 分支 |
| 单线程 CPU 瓶颈 | 多线程 + Send/Sync trait，**编译期防数据竞争** |
| 重构沉默退化 | 类型系统硬强制（没有 any / ignore 这种逃生口） |
| 性能 | 编译到原生机器码，无 GC，常态 5-50x JS |

**关键区别**：Rust 不是"工具好"，是"**编译器不让你犯错**"。TS 的所有上面这些痛点都有解，但都靠**自觉**。Rust 把它们做成**语言/编译器约束**。

---

## 4. 但 Rust 不是免费的

诚实地讲，Rust 给你这些好处的代价：

| 代价 | 程度 |
| --- | --- |
| **学习曲线** | 借用检查器、生命周期、trait bounds 让新人前 2 个月经常被编译器骂 |
| **开发速度** | 同一个功能初次实现，Rust 通常比 TS 慢 1.5-3 倍（之后维护更快） |
| **生态** | npm 600 万包，crates.io 15 万；金融/精算专门库 TS 多于 Rust |
| **招聘** | Rust 工程师比 TS 工程师少 10-20 倍且贵 |
| **团队适应期** | 2-3 个核心人 6 个月，全员 1 年起 |
| **业务编排不擅长** | 写 CRUD / API 路由 / 简单 ETL，Rust 比 TS 啰嗦 |

---

## 5. 适合"现在就开始"的场景

不是全切，是**精准切关键路径**。下面是高 ROI 的候选：

| 候选场景 | 为什么 ROI 高 |
| --- | --- |
| **保费计算核心引擎** | 精度 + 性能 + 正确性都是 Rust 强项 |
| **批量结算 / 对账** | CPU 密集，Node 单线程是瓶颈 |
| **风险模型 / 蒙特卡洛模拟** | 极致性能直接转化为业务能力（能跑更多场景） |
| **规则引擎（健康险核保规则、车险定价）** | 类型系统 + 模式匹配 + 错误处理三件套都用上 |
| **导出 / 报表生成 (大文件)** | Rust 写文件比 Node 快 5-10x，内存少 10x |

**不适合换**：
- 业务 API / 后台管理 / 前端 BFF → 继续 TS
- 团队没有任何人懂 Rust 又赶项目 → 别上
- 系统只有几千 QPS 又没精度问题 → 没必要

---

## 6. 一个真实的引入路径（推荐）

```
阶段 1（1-2 个月）: 1-2 人学 Rust 基础，做 PoC
  ├─ 挑一个最痛的纯计算函数（比如保费公式）
  ├─ 用 Rust 写一份，napi-rs 暴露给现有 Node 服务调用
  └─ 跑 A/B 对比：性能 / 内存 / 精度

阶段 2（3-6 个月）: 把所有"纯计算热点"迁到 Rust
  ├─ 团队 1-2 人专门写 Rust 模块
  ├─ TS 主仓库继续承担业务编排
  └─ 沉淀 Rust 风格规范、错误处理范式、测试套路

阶段 3（6-12 个月）: 评估全栈或独立服务
  ├─ 如果计算量大到值得 → 拆独立 Rust 服务
  ├─ 团队扩到 3-5 人懂 Rust
  └─ 不强求全栈，TS + Rust 并存是常态
```
