# 01 · 架构师的角色与心智

> 在动手学技术之前，先搞清楚架构师**到底是什么角色**、**对谁负责**、**用什么思维方式做决定**。这一章不写代码，但它决定了后面 15 章你学的东西能不能用对地方。

---

## 1. 架构师的几种典型形态

行业里"架构师"这个 title 含义混乱。常见的有四种，工作重心差别很大：

| 类型 | 关注层级 | 典型产出 | 典型组织里的归属 |
|---|---|---|---|
| **应用架构师 (Application Architect)** | 单个系统/产品 | 系统拆分、模块边界、技术选型 | 业务团队 |
| **解决方案架构师 (Solution Architect)** | 跨系统的端到端方案 | 集成方案、技术选型对比、PoC | 大客户 / 售前 / 项目 |
| **领域架构师 (Domain Architect)** | 某个业务域（支付、风控、库存） | 域内模型、能力地图、领域 API | 业务线技术中台 |
| **企业架构师 (Enterprise Architect)** | 整个公司技术资产 | 技术战略、IT 治理、标准规范 | CTO Office |

**自我定位是第一性问题。** 一个企业架构师拿应用架构师的标准评估，会显得"不接地气"；一个应用架构师拿企业架构师的标准评估，会显得"格局不够"。两者都在错位评估。

## 2. 架构师的核心职责（不分类型都要做）

### 2.1 做技术决策并对结果负责

- 选不选 K8s、用不用微服务、上不上某中间件 —— 决策本身不难，难的是**理由要经得起 6 个月后回顾**。
- "我们当时选 X 是因为 A、B、C 三个约束" 比 "我们用了 X" 重要得多。

### 2.2 控制系统的本质复杂度

- 软件复杂度有两种：**本质复杂度**（业务真实要求带来的）和 **附加复杂度**（架构/技术选型/抽象不当引入的）。
- 架构师 70% 的工作是**别让附加复杂度蔓延**。剩下 30% 是把本质复杂度装进合适的容器。

### 2.3 跨团队对齐

- 一个决策只有被相关团队**理解、认可、执行**才算落地。文档发出去不等于对齐。
- 对齐成本经常比技术方案本身更高。

### 2.4 建立"非功能"防线

- 业务需求关注**功能**（能不能下单），架构师关注**非功能**（千万 QPS 下能不能下单、出问题怎么恢复、加新业务多快能上）。
- 性能、可用性、可观测性、安全、可维护性、可演进性 —— 这些没有 PM 帮你提需求，全靠架构师主动设定标准。

### 2.5 输出可执行的资产

| 资产 | 作用 |
|---|---|
| **ADR (Architecture Decision Record)** | 把"为什么这么定"留给未来的人，详见 [14-soft-skills.md](14-soft-skills.md) |
| **技术方案文档** | 让一线工程师知道怎么落地 |
| **参考实现 (Reference Implementation)** | 比 100 页文档更有说服力的代码示范 |
| **架构图** | C4、UML、流程图，统一团队心智 |
| **技术规范** | API 设计规范、代码规范、可观测性规范 |

## 3. 三种关键思维方式

### 3.1 权衡思维（Trade-off Thinking）

> **架构无银弹。每个决策都是把 A 换成 B。讲不出"代价"的方案不是方案。**

经典例子：

| 选项 | 得到 | 失去 |
|---|---|---|
| 微服务 | 独立部署、技术异构、团队自治 | 运维复杂、网络延迟、分布式事务难 |
| 强一致 | 数据正确 | 可用性下降、延迟升高 |
| 同步调用 | 简单、好调试 | 雪崩风险、扩展性差 |
| 异步消息 | 解耦、削峰填谷 | 最终一致、调试难、顺序难保证 |
| 自建中间件 | 完全可控 | 长期人力黑洞 |
| 用云服务 | 快速、稳定 | 锁定、贵 |

**练习**：拿任何一个你正在做的技术决策，写出"得到"和"失去"两列。写不出 3 条"失去"就不算想清楚。

### 3.2 量化思维

> **没有数字，"高并发"、"低延迟"、"大数据量"都是噪音。**

每次说出形容词，立刻问自己/对方：
- "高并发" → 多少 QPS？峰值还是均值？
- "低延迟" → P99 多少 ms？端到端还是单跳？
- "大数据量" → 多少行？多大磁盘？日增长？

**基础数字感**（Jeff Dean 的"latency numbers every programmer should know"简化版，背下来）：

| 操作 | 量级 |
|---|---|
| L1 缓存访问 | 1 ns |
| 内存随机访问 | ~100 ns |
| 1KB 数据走万兆网 | ~10 μs |
| SSD 随机读 | ~150 μs |
| 同城数据中心往返 | ~500 μs |
| 机械硬盘随机读 | ~10 ms |
| 跨大洲网络往返 | ~150 ms |
| 一次完整 TCP+TLS 握手 | ~100 ms（公网） |

记住这套尺子，你就能在脑子里对"加个缓存能省多少"、"加个 RPC 调用要花多少"做秒级估算。

### 3.3 演进思维

> **架构不是设计出来的，是长出来的。**

- 不要为 5 年后的预想流量做今天的复杂设计 —— 90% 的预想流量永远不会来。
- 但要**为演进留接口**：模块边界清晰、数据耦合低、关键路径可替换。
- "**Make it work → Make it right → Make it fast**" 是顺序，倒过来就翻车。

## 4. 架构师的一天（典型时间分配）

不同公司差异巨大，但一个"健康"的应用/领域架构师大致是：

```
30%  写代码 / Review 关键路径代码
25%  和团队/PM/上游开评审会
20%  写文档（方案、ADR、规范）
15%  调研 / 做 PoC
10%  跨团队对齐与"修桥"
```

**警报信号**：
- 写代码时间 < 10% → 你正在脱离一线，决策会越来越虚。
- 写代码时间 > 50% → 你正在被当成高级工程师用，架构职能被淹没。
- 开会 > 50% → 协调成本失控，应该开始拆决策权而不是亲力亲为。

## 5. 反模式（要警惕的自己）

| 反模式 | 表现 | 为什么糟糕 |
|---|---|---|
| **PPT 架构师** | 只画图、不碰代码、不知细节 | 方案落地永远走样 |
| **新技术狂热** | 见啥用啥、动辄"重构"、爱搞中台 | 把简单问题搞复杂、把团队拖入泥潭 |
| **NIH (Not Invented Here)** | 一切自研、看不起开源/云服务 | 烧团队、慢迭代、留一堆没人维护的轮子 |
| **决策黑箱** | 决定一拍脑袋出 + 不写 ADR | 半年后没人知道为啥这么做、改不动 |
| **过度抽象** | 三层接口起步、留各种"扩展点" | 没人懂、没人用、删都不敢删 |
| **甩手掌柜** | 给方案不管落地 | 团队执行时遇坑没人兜，方案变形 |
