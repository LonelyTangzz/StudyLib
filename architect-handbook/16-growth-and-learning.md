# 16 · 成长路径与学习方法

> 架构师不是某一刻"升上去"的 title，是一种长期形成的判断力。这一章讲：怎么从工程师成长为架构师、怎么持续学习、怎么避开成长陷阱。

---

## 1. 成长阶段

### 1.1 一种参考分级（不绝对）

| 阶段 | 关键特征 | 通常年限 |
|---|---|---|
| **L1 - 新手工程师** | 在指导下完成清晰任务 | 0-2 年 |
| **L2 - 熟练工程师** | 独立完成模块、写得清楚 | 2-4 年 |
| **L3 - 资深工程师 (Senior)** | 设计中型系统、leading 项目 | 4-7 年 |
| **L4 - 技术专家 (Staff)** | 跨团队影响、解决组织级问题 | 7-12 年 |
| **L5 - 首席架构师 (Principal)** | 业务方向决策、技术战略 | 10+ 年 |
| **L6 - Distinguished / Fellow** | 行业级影响力 | 极少数 |

**反例**：年限够了 ≠ 在那个级别。L3 卡 10 年的人比比皆是。

### 1.2 工程师 → 架构师 的核心跃迁

| 维度 | 工程师 | 架构师 |
|---|---|---|
| 思考范围 | 模块 | 系统 → 业务 → 组织 |
| 时间维度 | 这个 sprint | 半年-3 年 |
| 衡量标准 | 我写了多少代码 | 我影响了多少决策 |
| 失败方式 | bug | 错误的方向 |
| 学习方式 | 学具体技术 | 学**判断**和**取舍** |

## 2. T 型 / π 型能力结构

### 2.1 T 型

横（广）+ 竖（深）：
- **竖**：至少一个领域深入到原理级（你的"金字招牌"）
- **横**：邻近领域能讨论、不被忽悠

### 2.2 π 型

两根竖（两个深入领域）+ 横。例如：
- 后端 + 数据
- 后端 + 安全
- 后端 + DevOps

资深架构师通常是 π 型。

### 2.3 反例：**I 型 / 一字型**

- **I 型**：只一根竖（一个领域很深，其他啥都不知道）→ 视野窄，做不了架构
- **一字型**：到处广但都不深 → 没真正"扎下"的领域，决策不被信任

## 3. 学习方法

### 3.1 学新技术的三层

| 层 | 怎么学 | 适用 |
|---|---|---|
| **会用** | 跟教程跑通 hello world | 评估 / 选型阶段 |
| **能讲** | 能给同事讲明白、能写文档 | 决定采用前 |
| **能改 / 能修** | 看源码、能贡献回去 / 能修 bug | 关键依赖 |

**反例**：所有技术都停在"会用"。架构师对**核心栈**必须到"能讲"甚至"能改"。

### 3.2 学新领域的套路

1. **找一本经典书**（不是博客）：读完，作笔记
2. **找一个项目跑通**：纸上谈兵不算
3. **学反对的观点**：每个技术都有人骂，看骂的内容能学到很多
4. **写下来**：自己讲一遍给团队/博客/笔记
5. **应用到工作**：实战是唯一的真正消化

### 3.3 阅读源码

- 选当前正在用的、核心的、看得懂的（不要上来就 Linux Kernel）
- 带着问题读（"它怎么实现 X？"）
- **画图** > 看代码：理解模块关系
- 每读懂一块写一段 200 字的笔记

### 3.4 学习的时间分配

工作日大头是业务。**架构师应该留至少 5 小时/周给"长期学习"**：
- 1 小时：行业动态（公司外发生了什么）
- 2 小时：深入读一本书 / 一篇 paper / 一段源码
- 2 小时：动手实践

不留时间，3 年内你会被新人甩开。

### 3.5 怎么读 paper

架构师该读的几类：
- **系统论文**：Google MapReduce、BigTable、Spanner；Amazon Dynamo；Facebook Memcache；现代如 RAMCloud、FaRM
- **数据库论文**：B+ Tree、LSM、MVCC、Raft、Spanner
- **分布式论文**：Lamport 系列、ZAB、Raft、CALM 定理、FLP
- **AI/ML 系统论文** (近几年)：Transformer 架构、LLM 系统设计

读 paper 的套路：摘要 → 引言 → 结论 → 图表 → 选感兴趣的章节细读。不必每篇全读。

## 4. 真实项目是最大老师

### 4.1 主动找有挑战的项目

- 大流量 / 高可用要求
- 数据量大 / 复杂业务
- 跨团队 / 跨技术栈
- 没人做过 / 不确定的方向

简单 CRUD 做 100 个，能力不会有跃迁。

### 4.2 故障是最好的学习

- 每次线上故障**至少花 2 小时**真的搞懂根因
- 写复盘，分享团队
- 故障复盘合集是你的私人金矿，定期回看

### 4.3 跨域轮换

如果有机会：
- 从业务后端 → 基础设施
- 从应用层 → 中间件
- 从单体 → 微服务 / 反过来
- 从开发 → SRE 一段时间

每次换给你 3-6 个月的"再当新手"的视角，能力会指数级长。

## 5. 写作与公开输出

### 5.1 为什么要写

- **逼自己想清楚**（说不出来就是没想明白）
- **复利效应**（一篇文章被读 N 次，比讲一次给 1 个人高效）
- **个人品牌**（公司外的人通过你的输出认识你）
- **吸引同类**（写得好的人会找上你 / 你能找到他们）

### 5.2 写什么

- 复盘文（事故、项目、决策）
- 技术深度文（一个机制的原理 + 你的实战经验）
- 选型对比（具体场景下 A vs B）
- 翻译/解读经典 paper / 源码
- **千万不要写"教程"**（你写的不如官方文档好），写**你的判断和经验**

### 5.3 输出渠道

- 团队内：周报、知识库、分享会
- 公司内：技术 blog、内部论坛
- 行业内：个人博客、GitHub、技术会议演讲

不必一开始就追求阅读量。**写给未来的自己**也是巨大收益。

---

## 6. 社群与同行

### 6.1 找对标人

- 在公司里找比你高 2 个级别的人，观察他们怎么想问题、怎么沟通、怎么决策
- 在公司外找你领域的"明星"（关注他们的 blog/talks/repo）
- 加入小而精的同行群（不是大水群）

### 6.2 给比要

- 帮人解卡点、答问题、写评论
- 时间长了，你的人脉就有了"利息"
- 一味索取的人会被疏远

## 7. 反模式与陷阱

### 7.1 技术陷阱

| 陷阱 | 表现 |
|---|---|
| **追新症** | 啥新追啥、什么都没沉淀 |
| **NIH 综合症** | 啥都自研、最后变成"轮子坟场维护员" |
| **架构占星术** | 画图很美、落地很差 |
| **过度抽象** | 把简单问题搞复杂 |
| **PPT 架构师** | 不碰代码、决策脱离实际 |
| **保守化** | 守着 5 年前的栈、看不上新东西、被时代抛弃 |

### 7.2 职业陷阱

| 陷阱 | 表现 |
|---|---|
| **被技术绑死** | 只懂某门技术，市场没需求就完蛋 |
| **只在小公司不出去看** | 视野窄，标准低 |
| **逃避做决策** | 永远"再调研一下"、错过窗口 |
| **不敢失败** | 不接挑战项目、不挑战权威 |
| **只技术不业务** | 永远是"工具" |
| **躺在 title 上** | Senior 了就停止精进 |

### 7.3 心态陷阱

| 陷阱 | 表现 |
|---|---|
| **冒名顶替综合症 (Imposter)** | "我配不上这个职位"，长期内耗 |
| **达克效应 (Dunning-Kruger)** | 学了点就觉得自己懂、不再深挖 |
| **舒适区** | 选熟悉的活，不挑战 |
| **比较焦虑** | 总看别人晋升快，不踏实做事 |

## 8. 平衡：技术深度 vs 业务理解 vs 人际能力

> **架构师的天花板由"最短的那块板"决定。**

很多 L3-L4 卡住的人，技术没问题，卡在沟通；很多 L4-L5 卡住的人，技术能力没问题、沟通也行，卡在不懂业务。

每年自检一次：
- 我技术上有进步吗？深 / 广？
- 我业务上更懂了吗？能讲清商业模式吗？
- 我人际影响力扩大了吗？跨团队推动事情更容易了吗？

三块都要补。**你忽视的那块，会决定你卡在哪一级**。

## 9. 一个粗略的 5 年路径建议

仅供参考，每个人节奏不同：

**Year 1-2（资深工程师起步期）**
- 把一项主技术钻到原理级
- 主导一个中型模块的设计与落地
- 开始写技术文档、ADR

**Year 3-4（资深 → 跨团队期）**
- 主导一个跨团队项目
- 深入一个邻近领域（数据 / 安全 / 基础设施）
- 开始公开输出（公司内分享、blog）
- 至少经历 1-2 次重大线上故障并主导复盘

**Year 5-7（架构师成型期）**
- 至少负责一次架构演进（单体 → 微服务、上云、数据架构升级）
- 跨业务团队推动技术标准
- 培养 1-2 个能接你的活的人
- 学一个完全陌生的领域（如果一直做后端，去碰前端 / 数据 / AI）
- 跟外部技术社区接触（会议、开源贡献）

**之后**
- 个人风格成型：你是"系统稳定型"、"业务驱动型"、"创新探索型"、"组织协调型"中的哪一类？
- 把自己变成"知识可传授"的状态：写书、做培训、贡献影响力

## 10. 给后期架构师的 7 条经验心法

1. **凡事问"为什么"3 次以上**，到根因再行动
2. **能用简单方案绝不上复杂的**，简单是最难的
3. **写下来**：决策、教训、判断 —— 不写就忘
4. **承认不知道**比假装知道有力得多
5. **量化一切**：数字让讨论高效
6. **照顾人**：技术由人推进、架构由人维护
7. **保持手感**：定期写代码，别变成只评审不动手的"领导"

## 12. 总结：本手册的精神

回到 [README.md](README.md) 的能力地图。读完 16 章你会发现一件事：

> **架构师不是"知道很多"，而是"能在不确定中做出好的取舍"。**

知识只是材料，**判断力**才是产物。判断力来自：
- 大量真实项目的反馈
- 对失败的诚实复盘
- 持续学习不停滞
- 跨人合作的磨练

希望这本手册成为你的脚手架，撑起你的成长，最终被你超越。
