# 09 · 性能工程

> "**先测量，再优化。**" 这句话每个架构师都听过，但 90% 的性能问题来自违反它。这一章给你测量、分析、优化、容量规划的完整方法论。

---

## 1. 性能的两个维度

| 维度 | 定义 | 关注 |
|---|---|---|
| **延迟 (Latency)** | 单次请求耗时 | 用户体验、SLA |
| **吞吐 (Throughput)** | 单位时间处理量 | 成本、容量 |

**关键认识**：它们不是同义词。低延迟 ≠ 高吞吐（缓存命中延迟 1ms，但单机吞吐受 CPU 限制）。高吞吐 ≠ 低延迟（流处理可能每秒百万条但单条延迟 100ms）。

---

## 2. 延迟为什么必须看分位数 (Percentile)

平均延迟是**最有欺骗性的指标**。99% 请求 10ms，1% 请求 10s，平均不到 100ms，但 1% 的用户体验是灾难。

**必看的分位数**：

| 指标 | 含义 | 关注度 |
|---|---|---|
| P50 (中位数) | 一半请求快于此 | 中 |
| P95 | 95% 快于此 | 高 |
| **P99** | 99% 快于此 | **核心** |
| P99.9 | 99.9% 快于此 | 高（大流量时） |
| Max | 最慢的一笔 | 用于发现极端值 |

**架构师定 SLO 用 P99，不用平均**。

### 2.1 长尾延迟 (Tail Latency) 放大效应

如果一个请求要打 100 个后端，每个后端 P99 = 100ms，则**整体 P99 ≈ 几百 ms**（你大概率会撞上至少一个慢的）。

后端越多，长尾被放大越严重。这就是为什么微服务调用链长了延迟变差。

缓解：
- **请求 hedging**（同时打两份，要一份就行）
- **超时积极**（宁可重试也不等长尾）
- 减少调用链深度

## 3. 性能分析方法论

### 3.1 USE 方法（Brendan Gregg，适合资源）

对每个资源（CPU、内存、磁盘、网络）问：
- **U**tilization：使用率多少？
- **S**aturation：是否饱和（排队）？
- **E**rror：有没有错误？

### 3.2 RED 方法（适合服务）

对每个服务问：
- **R**ate：每秒请求数
- **E**rrors：错误率
- **D**uration：延迟分布

### 3.3 自顶向下排查

```
1. 业务指标异常（订单 RT 高）
2. 看应用级监控（哪个接口慢？P99 高在哪个阶段？）
3. 看依赖（DB / Cache / 下游服务）
4. 看资源（CPU / Mem / IO / Net）
5. 进容器/进程内（profile、stack trace）
```

不要倒过来从资源开始猜。**先问业务现象，再下沉到机器。**

## 4. 性能分析工具

### 4.1 操作系统层

| 工具 | 看什么 |
|---|---|
| `top` / `htop` | CPU、内存进程级别使用 |
| `vmstat` | 系统级 CPU、内存、IO、swap |
| `iostat` | 磁盘 IO 详情 |
| `pidstat` | 单进程的 CPU、IO、上下文切换 |
| `mpstat` | 每个 CPU 核的使用 |
| `sar` | 历史指标查询 |
| `iftop` / `nethogs` | 网络带宽分进程查看 |

### 4.2 网络层

| 工具 | 看什么 |
|---|---|
| `ping` | 连通性 + 基础延迟 |
| `traceroute` / `mtr` | 链路每跳延迟 |
| `tcpdump` | 抓包 |
| `wireshark` | 协议分析 |
| `ss` / `netstat` | 连接状态、队列 |
| `tcprtt` (BCC) | TCP RTT 实测 |

### 4.3 应用层 Profiling

| 类型 | 工具 |
|---|---|
| **CPU Profiling** | Java: async-profiler, JFR; Go: pprof; Python: py-spy; Linux: perf + flamegraph |
| **Memory Profiling** | Java: JFR, MAT; Go: pprof; Python: tracemalloc |
| **Lock Profiling** | Java: JFR, jstack; Go: pprof mutex/block |
| **GC Analysis** | Java: GC log + GCViewer; Go: pprof + gctrace |
| **eBPF** | bcc, bpftrace（Linux 通用，无侵入） |

### 4.4 火焰图 (Flame Graph)

CPU 时间在哪儿花的可视化：
- X 轴 = 总采样占比（不是时间序列）
- Y 轴 = 调用栈深度
- 宽 = 占比高
- 看到**高且宽**的方块 = 性能热点

**第一次性能优化任务**：拿个生产负载跑 30 秒火焰图，找最宽的那块函数。十有八九能找到惊喜。

## 5. 常见优化套路

### 5.1 缓存

**第一性手段**。但要清楚加缓存的代价：
- 一致性窗口（接受脏数据多久）
- 失效复杂度（什么时候 invalidate）
- 内存成本
- 缓存被击穿/穿透/雪崩的兜底

不要无脑加。详见 [07](07-middleware-and-messaging.md)。

### 5.2 批量化 (Batching)

- 1000 次 `INSERT` → 1 次 `INSERT VALUES (...), (...), ...`
- 1000 次 HTTP 请求 → 1 次 POST batch
- 1000 次 Redis GET → MGET 一次

**收益常常是 100x**，因为每次往返开销（网络 + 反序列化）固定，批量摊薄。

### 5.3 异步化

把"必须串行等"改成"发出即返回"：
- 用户下单成功后，发邮件/写日志/打点 → 全部异步
- 写主流程要返回响应；后处理走 MQ

### 5.4 并行 / 并发

- 多个独立的下游调用 → 并发发出，等齐就行（不要串行）
- CPU 密集 → 多线程 / 多核
- IO 密集 → 异步 IO / 协程

### 5.5 减少数据传输

- 字段裁剪（不要 `SELECT *`、不要全字段返回 JSON）
- 压缩（HTTP gzip/br、gRPC 压缩、Protobuf 比 JSON 小）
- 二进制协议（Protobuf、MessagePack、FlatBuffers）
- 增量传输（CDC、delta 推送）

### 5.6 连接池

DB / HTTP / Redis 都要用**长连接 + 池化**，每次建连成本（特别是 TLS）能吃掉一半性能。

### 5.7 数据库优化

- 加合适的索引
- 用 `EXPLAIN` 验证执行计划
- 避免 N+1（用 JOIN 或 IN 一次取）
- 长事务拆短
- 大事务分批
- 读写分离
- 分库分表（最后手段）

### 5.8 JVM / GC 调优（Java 系）

- 选合适的 GC：G1 / ZGC / Shenandoah
- 堆大小：不是越大越好，大堆 → 长 GC
- 关注 GC 暂停时间分位数，不只是平均
- 监控 `Full GC` 频率，触发就该报警

## 6. 容量规划

### 6.1 一个简化但实用的模型

```
所需容量 = 业务峰值 QPS × 安全系数 / 单实例承载能力
```

例：业务峰值 10k QPS，安全系数 2x，压测单实例 500 QPS，则需要 40 个实例。

### 6.2 全链路容量

不只算应用层。把链路每一跳都列出来：

```
[用户] → [CDN/网关] → [应用] → [Redis] → [DB主] → [DB从]
                              → [Kafka] → [下游消费]
```

每一跳都要算容量，**瓶颈在最弱的那一跳**。常见瓶颈是 DB 写、Redis 单 key、MQ 单分区。

### 6.3 容量评估的常见陷阱

- 只测**单接口**：实际是多接口混合负载。
- 没考虑**峰均比**：电商日均 1k QPS，秒杀峰值 100k QPS。
- 没考虑**冷启动**：扩容后实例缓存空，反而更慢。
- 没考虑**依赖容量**：自己加机器，DB 连接池打满。

## 7. 压测 (Load Testing)

### 7.1 类型

| 类型 | 目的 |
|---|---|
| **基线压测** | 确认正常负载下的 RT/QPS |
| **峰值压测** | 模拟业务峰值，验证容量 |
| **极限压测 (Stress)** | 持续加压直到崩，找瓶颈 |
| **耐久压测 (Soak)** | 长时间持续，发现内存泄漏、连接泄漏 |
| **混沌压测** | 加压同时随机杀实例 / 断网 |

### 7.2 工具

- **wrk / wrk2**：HTTP 简单高效
- **JMeter**：图形化、协议多、重
- **k6**：脚本友好（JS）、报表好
- **Gatling**：Scala 写、并发模型好
- **Locust**：Python 脚本
- **ghz**：gRPC 压测
- **vegeta**：Go 写，CLI 友好

### 7.3 压测注意

- **不要打生产**（除非确定能承住且对业务无影响）
- **压测环境配置 ≈ 生产**（按比例缩，关键中间件要真）
- **数据要真实**（量 + 分布），别用 1k 行的小表压测
- **关注全链路指标**，不只是压测端 RT
- **暴力打高 QPS 看 RT** 和 **固定 RT 找最高 QPS** 是两种思路，按场景选

## 8. 性能问题排查 Checklist

线上慢/挂时按这个序：

1. **是不是刚发版/变更？** → 立刻回滚 / 关变更，先恢复再分析
2. **流量异常？** → 看 QPS 曲线，是不是突然涨了
3. **依赖异常？** → DB / Redis / 下游服务监控
4. **资源异常？** → CPU / 内存 / IO / 连接数
5. **GC？** → Java 系看 GC 频率和暂停
6. **慢日志 / 慢 SQL？**
7. **链路追踪看耗时分布** → 慢在哪一跳
8. **代码层 profile** → 找具体函数

## 9. 反模式

| 反模式 | 痛点 |
|---|---|
| 优化没测量数据 | 优化错地方 |
| 用平均延迟看体验 | P99 灾难被掩盖 |
| 无脑加缓存 | 引入一致性问题 |
| 串行调用所有下游 | 长尾叠加 |
| 大事务长事务 | DB 锁等、复制延迟 |
| 不做容量规划等出事 | 永远在救火 |
| 压测打不动还宣称没问题 | 压测端是瓶颈 |
| 没有连接池 | 每次新建连接 |
