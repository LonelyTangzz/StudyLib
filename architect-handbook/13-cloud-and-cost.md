# 13 · 云架构与成本治理

> 云让"几小时建一个数据中心"成为可能，也让"一夜烧掉 10 万美金"成为可能。架构师必须既懂**云服务模型**，也懂**FinOps 成本治理**。

---

## 1. 云服务模型

### 1.1 三层抽象

| 模型 | 你管什么 | 云管什么 | 例 |
|---|---|---|---|
| **IaaS** | OS、运行时、应用 | 物理机、网络、虚拟化 | EC2、ECS（虚拟机）、VPC |
| **PaaS** | 应用、数据 | OS、运行时、扩缩容 | App Engine、RDS、ElasticBeanstalk |
| **SaaS** | 配置 + 用 | 全部 | Gmail、Salesforce、Notion |

新增的：

| 模型 | 描述 | 例 |
|---|---|---|
| **CaaS (Container as a Service)** | 容器调度 | ECS、GKE、AKS |
| **FaaS (Function as a Service)** | 无服务器、按请求计费 | Lambda、Cloud Functions、Cloudflare Workers |
| **DBaaS** | 托管数据库 | RDS、DynamoDB、Aurora |

### 1.2 选型层级原则

> **能用 SaaS 不写 PaaS，能用 PaaS 不上 IaaS。**

例外：
- 极致性能 / 特殊硬件需求 → IaaS
- 强合规 / 数据主权 → 私有云 / 自建
- 极致成本（大规模稳态负载） → 自建经常更便宜

## 2. 三大公有云速判

| 维度 | AWS | Azure | GCP |
|---|---|---|---|
| 市场份额 | 最大 | 第二 | 第三 |
| 服务广度 | 最广 | 强（企业、Office 生态） | 数据/AI 强 |
| Kubernetes | EKS | AKS | GKE（最早成熟） |
| 数据仓库 | Redshift | Synapse | BigQuery |
| Serverless | Lambda | Functions | Cloud Functions, Cloud Run |

国内：阿里云、腾讯云、华为云。生态上阿里最大、华为合规友好。

## 3. 云上核心概念

### 3.1 Region / AZ / Edge

| 层级 | 含义 |
|---|---|
| **Region** | 一个地理位置（如 us-east-1, ap-east-1） |
| **Availability Zone (AZ)** | Region 内的独立机房，通常 3 个 |
| **Edge / PoP** | CDN 边缘节点，几百个 |

- AZ 间网络延迟 < 2ms
- Region 间延迟 50ms+
- **生产架构默认多 AZ**，单 AZ 不可靠（AWS 历史上多次单 AZ 大故障）
- **核心业务上多 Region**，但复杂度大幅上升

### 3.2 VPC / 网络

- **VPC**：你自己的虚拟网络，控制 CIDR、子网、路由
- **公有子网 / 私有子网**：是否能直接访问公网
- **NAT Gateway**：私有子网通过它出网
- **VPC Peering / Transit Gateway**：跨 VPC 连通
- **Direct Connect / Express Route**：专线接入

**架构原则**：业务服务在私有子网、只暴露必要端口、外网入口靠 LB + WAF。

### 3.3 计算服务全景

| 类型 | AWS | 适用 |
|---|---|---|
| 虚拟机 | EC2 | 通用、传统 |
| 容器（自管 K8s） | EC2 + K8s | 完全控制 |
| 容器（托管 K8s） | EKS | K8s 但不管控制面 |
| 容器（无服务器） | Fargate, Cloud Run | 容器 + 按请求计费 |
| Serverless 函数 | Lambda | 事件驱动、短任务 |
| 边缘计算 | Lambda@Edge, CloudFront Functions, Cloudflare Workers | 极低延迟、全球部署 |

### 3.4 存储服务全景

| 类型 | AWS | 用途 |
|---|---|---|
| 对象存储 | S3 | 文件、备份、静态资源 |
| 块存储 | EBS | VM 磁盘 |
| 文件存储 | EFS, FSx | 共享文件系统 |
| 归档 | S3 Glacier | 极便宜、读取慢 |
| 数据库 | RDS, Aurora, DynamoDB, Redshift, ... | 见 [06](06-data-architecture.md) |

**S3 是云上最便宜、最可靠的存储**，11 个 9 的持久性。一切静态资源、备份、日志归档默认走 S3。

## 4. 云原生 (Cloud Native)

### 4.1 CNCF 定义的关键技术

- 容器化
- 微服务
- 声明式 API
- 不可变基础设施
- 服务网格
- 可观测性

### 4.2 12-Factor App 原则（云上应用基本盘）

1. **代码库** —— 一个应用一份代码库
2. **依赖** —— 显式声明、隔离
3. **配置** —— 环境变量
4. **后端服务** —— 视为附加资源（DB / Cache 都可换）
5. **构建、发布、运行** —— 严格分离
6. **进程** —— 无状态、share-nothing
7. **端口绑定** —— 自包含、绑定端口对外服务
8. **并发** —— 通过进程模型扩展
9. **易处理** —— 快启动、优雅停机
10. **开发-生产等价** —— 各环境尽量一致
11. **日志** —— 当作事件流，写 stdout
12. **管理进程** —— 一次性任务（migration）也是代码

这 12 条是云上跑应用的 baseline，**违反任何一条都会在某个点撞痛**。

## 5. 多区域 / 全球架构

### 5.1 何时需要

- 用户全球分布（延迟）
- 数据主权要求（数据不能出境）
- 灾备（单 region 整个挂）

### 5.2 难点

- **数据复制延迟**：跨大洲 100-300ms RTT，强一致几乎不可能
- **写冲突解决**：多写场景必须有冲突策略
- **流量调度**：DNS / GeoDNS / Anycast 路由用户到最近 region
- **运维复杂度**：监控、发版、配置都要跨 region 协调

### 5.3 模式

| 模式 | 描述 |
|---|---|
| **Active-Passive** | 一个主 Region，其他备用；故障时切换 |
| **Active-Active** | 多 Region 同时服务，数据双向同步 |
| **单元化 (Cell)** | 用户路由到固定单元，单元内自包含 |

阿里"异地多活"基本是单元化。

## 6. Serverless

### 6.1 价值

- **零运维**：服务器你不管
- **极致弹性**：0 → 万并发自动扩
- **按用付费**：不调用不花钱

### 6.2 限制

- **冷启动**（首次调用慢，依赖大时更明显）
- **执行时长上限**（Lambda 15 分钟）
- **状态管理难**（无状态、状态外移）
- **本地调试难**
- **供应商锁定**

### 6.3 适用 / 不适用

| 适合 | 不适合 |
|---|---|
| 事件驱动短任务（上传图片处理、Webhook） | 长跑任务 |
| 间歇性负载 | 持续高负载（成本反而高） |
| API 后端（中低流量） | 严格低延迟 |
| 批处理 / Cron | 数据库连接密集（容易把 DB 打挂） |

### 6.4 Serverless 数据库

- **DynamoDB**：按请求计费、自动扩
- **Aurora Serverless**：按需扩缩
- **Neon / PlanetScale**：现代 Serverless Postgres / MySQL

## 7. 成本治理 (FinOps)

云账单的失控是常态。**架构师必须把成本当一等公民**。

### 7.1 三大原则

1. **可见 (Visibility)**：每个团队/服务的成本能算清
2. **优化 (Optimization)**：定期找浪费、优化
3. **治理 (Governance)**：成本预算、超支告警、审批

### 7.2 成本归因

- **Tagging 规范**：所有资源打 owner / project / env 标签
- **成本报表**按维度切片
- **Showback / Chargeback**：把成本透明给团队，激励优化

### 7.3 常见浪费源

| 浪费 | 怎么找 |
|---|---|
| **僵尸资源** | 没人用的实例、未挂载的磁盘、闲置弹性 IP |
| **过度供应** | CPU 使用率长期 < 20% 的实例 |
| **存储不分层** | S3 标准存了一年没动的数据 |
| **流量贵在出口** | 跨 AZ / 跨 Region / 出公网都贵 |
| **托管服务用错型号** | RDS 选了 io2 实际不需要 |
| **测试环境忘关** | 周末空跑 |
| **日志失控** | CloudWatch / 日志成本爆炸 |

### 7.4 优化手段

#### 7.4.1 实例

- **Right-sizing**：按实际负载选规格
- **Reserved Instance / Savings Plan**：长期承诺，省 30-70%
- **Spot Instance**：可中断的工作（批处理、可重试任务），省 70-90%
- **Auto Scaling**：按负载弹性

#### 7.4.2 存储

- **S3 生命周期策略**：热 → IA → Glacier 自动转
- **压缩**（gzip、zstd、Parquet）
- **TTL**：日志、备份过期自动删

#### 7.4.3 网络

- **同 AZ 通信**：避免跨 AZ 数据流量费
- **PrivateLink / VPC Endpoint**：内网直连云服务，省出口费
- **CDN**：静态资源走 CDN 比 S3 直出便宜（CDN 出口便宜）

#### 7.4.4 数据库

- 读写分离、缓存先扛
- 大表归档到便宜存储
- 选合适的引擎（DynamoDB vs Aurora 成本结构差极大）

### 7.5 FinOps 工具

- 云厂自带：AWS Cost Explorer、GCP Billing、Azure Cost Management
- 第三方：CloudHealth、Apptio Cloudability、Vantage
- 开源：OpenCost (K8s 成本归因)

## 8. 锁定 (Vendor Lock-in)

### 8.1 锁定的几个层级

| 层级 | 例 | 解锁难度 |
|---|---|---|
| **基础设施** | EC2 / VPC | 中（K8s 抽象后还好） |
| **数据库** | DynamoDB、BigQuery 专有 SQL 方言 | 高 |
| **消息** | SQS / SNS / Pub/Sub | 中 |
| **专有服务** | Lambda 触发器、Cognito、Athena | 极高 |
| **数据迁移成本** | PB 级数据出云 | 极高（带宽费 + 时间） |

### 8.2 多云策略

| 策略 | 利弊 |
|---|---|
| **单云专注** | 用尽云的能力，效率高；但风险集中、议价权差 |
| **多云被动** | 不同业务用不同云 | 不锁定但管理复杂 |
| **多云主动 (抽象层)** | 自研抽象层屏蔽云差异 | 抽象层成本可能比锁定成本还高 |

**实践经验**：除非有强合规 / 议价 / 历史原因，**单云 + 关键数据可迁移** 是最优解。强行多云常常两头不讨好。

## 9. 反模式

| 反模式 | 痛点 |
|---|---|
| 把 IaaS 当 PaaS 用 | 浪费云能力 |
| 单 AZ 部署 | 一次故障全挂 |
| 不打 tag | 成本归不到团队 |
| 闲置资源不清 | 月底账单震惊 |
| Lambda 跑长任务 | 超时 / 重试灾难 |
| 跨 Region 同步只看延迟 | 忽略流量成本 |
| 多云就为了"不锁定" | 抽象成本 > 锁定成本 |
| 用云就不考虑代码效率 | "反正能弹"，账单爆 |
| 把 S3 当文件系统用 | 频繁小请求 = 高账单 |
