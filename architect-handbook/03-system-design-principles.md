# 03 · 系统设计原则

> 系统设计不是画框架图。是一组**可以被验证**的原则：可扩展性、高可用性、一致性、性能、可维护性。这一章把这些"形容词"翻译成可量化、可决策的语言。

---

## 1. 可扩展性 (Scalability)

### 1.1 两个方向

| 方向 | 含义 | 上限 |
|---|---|---|
| **垂直扩展 (Scale Up)** | 加大单机配置（更大 CPU/内存/磁盘） | 物理 + 价格指数级 |
| **水平扩展 (Scale Out)** | 加机器、加节点 | 理论无限，受架构限制 |

**架构师默认假设**：长期看必须支持水平扩展。但短期可以靠垂直扩展买时间。**先垂直、再水平**是常见的成本最优路径。

### 1.2 阻碍水平扩展的常见模式

| 模式 | 表现 | 破解 |
|---|---|---|
| **有状态服务** | 用户必须打到同一台机 | 把状态外移到 Redis/DB |
| **单点数据库** | 加机器但都打一个 DB | 读写分离 → 分库分表 |
| **全局锁 / 全局序列** | 加机器但并发线性受限 | 分段、雪花 ID、本地序列 |
| **跨机事务** | 节点越多越慢越易冲突 | Saga / TCC / 最终一致 |
| **fan-out 查询** | 一次请求查所有分片 | 路由层只查一片 |

### 1.3 可扩展性的量化语言

- "支持 10x 用户"是空话。
- "当前 1k QPS 写、20k QPS 读、P99 < 100ms；目标半年后 5k 写、100k 读、P99 不变"是可设计的命题。

### 1.4 容量规划基本套路

1. 找出关键链路（钱、用户体验最敏感的路径）。
2. 测出单实例上限（吞吐 + 资源使用）。
3. 算出业务峰值需要多少实例，乘以**安全系数 2–3 倍**。
4. 算出依赖项（DB、缓存、MQ）能不能承住，往往**瓶颈不在应用层**。

## 2. 高可用性 (High Availability)

### 2.1 SLA / SLO / SLI

| 缩写 | 含义 | 谁用 |
|---|---|---|
| **SLI (Indicator)** | 实际测出来的指标（成功率、延迟） | 工程师 |
| **SLO (Objective)** | 目标值（"99.9% 请求 < 200ms"） | 团队内 |
| **SLA (Agreement)** | 写给客户/合同的承诺 + 违约赔付 | 商务 |

通常 **SLA < SLO < SLI 实测值**，留出 buffer。

### 2.2 "几个 9" 的现实含义

| 可用性 | 全年宕机时长 | 每月 |
|---|---|---|
| 99% | 87.6 小时 | 7.3 小时 |
| 99.9% | 8.76 小时 | 43.8 分钟 |
| 99.95% | 4.38 小时 | 21.9 分钟 |
| 99.99% | 52.6 分钟 | 4.38 分钟 |
| 99.999% | 5.26 分钟 | 26 秒 |

**架构含义**：每多一个 9，成本和复杂度大约是 10x。**不要盲目追 5 个 9**，先问业务真的需要吗。

### 2.3 提升可用性的手段（从便宜到贵）

1. **冗余**：N+1 实例，单台挂不影响。
2. **健康检查 + 自动剔除**：负载均衡器/注册中心识别死实例。
3. **超时 + 重试 + 断路器 (Circuit Breaker)**：防止级联失败。
4. **限流 + 降级**：保住主功能，主动牺牲非关键功能。
5. **隔离 (Bulkhead)**：线程池/连接池按业务划分，互相不拖累。
6. **多可用区 (AZ)**：单机房挂还能跑。
7. **多地域 (Region)**：单地域大灾难还能跑。
8. **多活 / 异地多活**：复杂度极高，电商/支付级别才需要。

### 2.4 雪崩与防雪崩

**典型雪崩链**：
```
下游慢 → 上游线程池占满 → 上游也慢 → 再上游线程池也占满 → 全链路挂
```

防雪崩三件套：
- **超时**（必须设，且分层设，外层 > 内层）
- **熔断**（短时间内错误率超阈值就快速失败，避免堆积）
- **限流**（保护自己，对来不及处理的直接 reject）

补充：
- **舱壁 (Bulkhead)**：不同业务用不同资源池，A 业务挂不连累 B。
- **背压 (Backpressure)**：下游告诉上游"我处理不过来"，上游主动放慢，不是无脑堆缓冲。

## 3. 一致性 (Consistency)

### 3.1 几种一致性模型（从强到弱）

| 模型 | 含义 | 代表 |
|---|---|---|
| **线性一致 (Linearizable)** | 全局一个时间线，写完立刻能读到 | etcd、ZooKeeper |
| **顺序一致 (Sequential)** | 每个客户端看到的顺序一致，但全局不强求 | 弱化的分布式存储 |
| **因果一致 (Causal)** | 有因果关系的操作顺序保持 | COPS、社交时间线 |
| **读你所写 (Read-your-writes)** | 同一用户写完能立刻读到 | 多数业务"够用"的最低线 |
| **单调读 (Monotonic Read)** | 不会越读越旧 | 读副本场景需保证 |
| **最终一致 (Eventual)** | 不再更新最终会收敛 | DynamoDB、Cassandra 默认 |

### 3.2 CAP 定理

> **Consistency, Availability, Partition tolerance —— 三者在网络分区发生时只能保两个。**

实际工程中：
- 分区**一定会发生**（机房断网、网卡抖动），所以 P 是必选。
- 真正的取舍是 **CP (强一致，牺牲可用性)** vs **AP (高可用，最终一致)**。

| 选 CP | 选 AP |
|---|---|
| 金融账务、库存扣减、配置中心 | 用户档案、商品详情、社交点赞 |
| etcd、ZooKeeper、Spanner | Cassandra、DynamoDB、Riak |

### 3.3 PACELC（CAP 的进化版）

> **如果有分区 P，要在 A 和 C 之间选；否则 (E)lse 在 L (Latency) 和 C 之间选。**

意思：即使没出分区，强一致也会比弱一致**慢**（要等多节点确认）。所以选 CP 系统时延迟天花板会被锁死。

### 3.4 BASE

NoSQL 圈子的口号：**B**asically **A**vailable, **S**oft state, **E**ventually consistent。

不是另一套定理，是"我们承认弱一致，请业务自己处理冲突"的承诺。

### 3.5 一致性的"够用"标准

实际项目里 99% 的"一致性需求"其实是**读你所写 + 单调读**。

例：
- 用户改了头像，他自己刷新立刻看到新的 = 读你所写 ✓
- 别的用户慢半秒看到新头像 = 完全可以接受 ✓

为这种场景上 etcd/Spanner 级别的强一致，是杀鸡用牛刀。

## 4. 幂等性 (Idempotency)

> **同一操作执行 N 次 = 执行 1 次的效果。**

**为什么对架构师极其重要**：分布式系统里**重试无法避免**（网络抖、超时、消息重投），不幂等就是数据灾难（重复扣款、重复下单）。

### 4.1 常见幂等设计

| 操作 | 幂等做法 |
|---|---|
| HTTP POST 创建订单 | 客户端生成幂等键 (idempotency_key)，服务端用唯一索引去重 |
| 扣库存 | 用版本号 / CAS：`UPDATE WHERE version=?` |
| 转账 | 流水表 + 唯一约束 (transfer_id) |
| 消息消费 | 消费表记录 message_id，先查再处理 |

### 4.2 幂等键的选择

- 业务键（订单号、流水号）天然带幂等含义。
- 没有业务键时，客户端必须生成 UUID 并在重试时复用。**服务端自己生成的 ID 不能用来幂等**——客户端重试时拿不到原 ID。

## 5. 弹性 (Resilience) 模式

### 5.1 重试 (Retry)

- **必须配退避** (exponential backoff + jitter)，否则重试风暴。
- **只能重试幂等操作**。
- **设最大次数**，到了直接失败，不要无限重试。
- 区分**可重试**（超时、5xx）和**不可重试**（参数错误、权限不足）。

### 5.2 超时 (Timeout)

- **每一层都要设**，且**外层 > 内层之和**。否则外层已 timeout，内层还在干，浪费资源。
- 默认值往往太长（HTTP 客户端默认无限），必须显式设置。

### 5.3 熔断 (Circuit Breaker)

三态：**Closed → Open → Half-Open**
- Closed：正常调用。
- 错误率超阈值 → 切到 Open，直接快速失败一段时间。
- 等待期结束 → 切到 Half-Open，放少量请求试探，成功则回 Closed，失败则继续 Open。

### 5.4 限流 (Rate Limiting)

| 算法 | 特点 |
|---|---|
| 计数器 | 简单，临界点流量翻倍 |
| 滑动窗口 | 平滑些，仍有突发 |
| **令牌桶 (Token Bucket)** | 允许短时突发、长期受限 |
| **漏桶 (Leaky Bucket)** | 强制平滑输出，不允许突发 |

业务接口通常用令牌桶（允许小突发更友好）；外部 API 网关常用漏桶（保护下游）。

### 5.5 降级 (Degradation)

- **自动降级**：触发某条件（错误率、依赖挂）自动切到 fallback。
- **手动降级**：开关推一下就关闭某功能。
- 设计时就要想好"不可用时显示什么"——空白页 vs 缓存数据 vs 简化版功能。

## 6. 可维护性 (Maintainability)

被低估的"非功能需求"。指标：

- **新人 onboard 时长**：从入职到能改一个 bug 用几天？
- **修改局部代价**：改一个字段，影响几个文件、几个团队？
- **回滚成本**：上线后发现问题，能否 5 分钟内回滚？
- **可调试性**：线上挂了，能否在不重启的情况下打出现场？

架构上的关键投入：
- 模块边界清晰（[05-architecture-patterns.md](05-architecture-patterns.md)）
- 接口稳定（[08-api-and-integration.md](08-api-and-integration.md)）
- 可观测性 (Observability) 完备（[11-observability.md](11-observability.md)）

## 7. 反模式

| 反模式 | 为什么糟 |
|---|---|
| "我们要 5 个 9" 但没人测 SLI | 数字是想象的，做不到 |
| 加机器就完事 | 数据库/缓存/MQ 才是真瓶颈 |
| 重试不设退避 | 雪崩制造机 |
| 超时全默认 | 几小时的超时等于没超时 |
| 强一致全场景 | 性能崩、成本爆 |
| 自动降级没人 review | 偷偷降级，业务挂了不知道 |
