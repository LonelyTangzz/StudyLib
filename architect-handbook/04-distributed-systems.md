# 04 · 分布式系统

> 上一章讲"应该是怎样"（原则），这一章讲"做不到怎么办"（机制）。架构师的硬核区。

---

## 1. 为什么分布式难

> "**The Network is Reliable.**" —— 这是分布式 8 大谬误之首。

分布式系统的难，根源在 8 件不能假设的事（Peter Deutsch 总结）：

1. 网络可靠
2. 延迟为零
3. 带宽无限
4. 网络安全
5. 拓扑稳定
6. 只有一个管理员
7. 传输成本为零
8. 网络是同构的

每条都被违反过，且经常同时违反。**架构师的工作就是在这些不可靠之上构建可靠的服务。**

## 2. 复制 (Replication)

### 2.1 为什么要复制

- **可用性**：一份挂了还有备份。
- **读扩展**：读副本分流。
- **就近访问**：副本放在离用户近的地方。

### 2.2 三种复制策略

| 策略 | 写延迟 | 数据丢失风险 | 典型 |
|---|---|---|---|
| **同步复制** | 高（等所有副本） | 0 | 金融交易 |
| **异步复制** | 低（写主就返回） | 可能丢一个窗口的数据 | MySQL 主从默认 |
| **半同步复制** | 中（等至少 1 个副本） | 比异步小 | MySQL 半同步插件、Kafka acks=1 |

### 2.3 复制拓扑

- **单主 (Single Leader)**：写都打主，读可分散。简单，但主是单点。
- **多主 (Multi Leader)**：多个写入点，常用于跨地域。**冲突解决**是难点。
- **无主 (Leaderless)**：客户端写多副本，读取多副本投票。代表：Dynamo、Cassandra。

### 2.4 复制延迟带来的问题

- **读到旧数据**：读副本还没同步到。
- **不一致的读**：第一次读到新，第二次读到旧。
- **跨用户视图错乱**：A 看到的消息列表，B 看不到。

解决：
- 关键查询走主库
- 用版本号/时间戳实现"读你所写"
- session 内固定路由到同一副本（实现"单调读"）

## 3. 分区 (Partitioning / Sharding)

### 3.1 分区策略

| 策略 | 优点 | 缺点 |
|---|---|---|
| **范围分区** | 范围查询快、有序 | 容易热点（最新数据集中） |
| **哈希分区** | 数据均匀 | 范围查询要扫所有分片 |
| **一致性哈希** | 节点增删时数据迁移少 | 仍可能热点 |
| **业务键分区** | 同租户数据在一起 | 大客户成超级热点 |

### 3.2 二级索引

分区表的二级索引有两种实现：
- **本地索引**：每个分片有自己的索引，查询要扇出到所有分片（scatter-gather）。
- **全局索引**：单独维护一个索引表，写要跨分片，但查只扫一片。

**架构含义**：分库分表后，绝大多数痛苦来自二级索引和跨分片 JOIN。设计时**让查询模式驱动分片键**，而不是反过来。

### 3.3 再平衡 (Rebalancing)

加节点 / 减节点时，数据要重新分配。三种方式：
- **固定分区数**（预先分 1000 个虚拟分片，按需挪）—— Riak、Elasticsearch
- **动态分区**（按 size 自动 split/merge）—— HBase、MongoDB
- **按节点数比例**（一致性哈希）—— Cassandra

## 4. 共识 (Consensus)

### 4.1 问题

> 一群节点，可能崩溃、网络可能分区，怎么让它们**就一个值达成一致**？

这是分布式系统的"祖问题"。所有"选主"、"分布式锁"、"配置同步"都是它的应用。

### 4.2 FLP 不可能性

异步网络下，**只要可能有一个节点崩溃，就不可能有确定性算法既保证安全又保证活性**。所以实际算法都做了取舍：依赖超时（牺牲一点严格性换可用）。

### 4.3 常见共识算法

| 算法 | 特点 |
|---|---|
| **Paxos** | 经典但难懂、难实现 |
| **Raft** | 为可理解性设计，事实标准，etcd / Consul / TiKV 都用 |
| **ZAB** | ZooKeeper 用的，类似 Paxos |
| **PBFT 系** | 容忍拜占庭节点（恶意/伪造）的共识，区块链常用 |

### 4.4 Raft 的三件事

1. **Leader Election**：超时没收到 leader 心跳就发起选举，多数派投票产生 leader。
2. **Log Replication**：所有写都经 leader，复制到多数派才 commit。
3. **Safety**：保证 commit 的 log 不会被覆盖。

**架构师必懂的结论**：
- 一个共识集群通常 3 / 5 / 7 个节点（奇数，多数派友好）。
- 容错能力 = `(N-1)/2`，5 节点能容 2 节点挂。
- **大集群不快**：节点越多，多数派越大，写延迟越高。共识集群通常 ≤ 7。

### 4.5 架构含义

- 任何"分布式锁"、"主选举"、"分布式配置"都最终落到一个共识系统（etcd / ZooKeeper）。
- **不要自己实现共识**。用现成的。

## 5. 分布式事务

### 5.1 为什么是难题

单机数据库给你 ACID，跨服务/跨库后，原子性不再天然成立。

**例**：转账，A 扣钱、B 加钱，跨两个服务。一边成功一边失败 = 数据不一致。

### 5.2 几种方案

#### 5.2.1 2PC (Two-Phase Commit)

```
Coordinator → [prepare] → all participants
Coordinator ← [yes/no]  ← all participants
Coordinator → [commit/abort] → all participants
```

**问题**：
- Coordinator 是单点，挂了所有 participant 锁住资源。
- 同步阻塞，性能差。

实际很少用在业务事务，多用于分布式数据库内部。

#### 5.2.2 3PC

加了一阶段降低阻塞概率，更复杂、仍有问题，工程少用。

#### 5.2.3 TCC (Try-Confirm-Cancel)

业务层实现两阶段：
- **Try**：预留资源（冻结额度）。
- **Confirm**：确认（扣减冻结额度）。
- **Cancel**：取消（释放冻结额度）。

需要业务接口都按这个模式拆，侵入大。但比 2PC 性能好、可控。

#### 5.2.4 Saga

把长事务拆成多个本地事务，**每个步骤配一个补偿动作**。出错时按反向顺序执行补偿。

```
T1 → T2 → T3 → T4 (出错)
         ↓
         C3 → C2 → C1 (补偿)
```

特点：
- 最终一致。
- 必须保证补偿幂等。
- 不能保证隔离（中间状态对外可见）。

#### 5.2.5 本地消息表 / Outbox Pattern

业务表和消息表在**同一个 DB 同一个事务**里写，保证"业务成功 = 消息一定发出"。

```
本地事务:
  INSERT order;
  INSERT outbox_message;
COMMIT;

后台进程: 轮询 outbox_message → 发 MQ → 标记已发
```

最可靠、最常用。详见 [07-middleware-and-messaging.md](07-middleware-and-messaging.md)。

### 5.3 选型建议

| 场景 | 推荐 |
|---|---|
| 强一致 + 性能不敏感 | TCC |
| 长流程（订单、签约） | Saga |
| 跨服务发消息 | Outbox |
| 同一服务 + 多 DB | XA 或避免（重新设计领域） |

**最佳策略经常是"避免分布式事务"**——重新拆分领域边界，把强一致需求收敛在一个服务/一个数据库里。

## 6. 时间与顺序

### 6.1 物理时钟的问题

- 服务器之间时钟不同步（NTP 也有偏差，毫秒级）。
- 闰秒、虚拟机迁移可能让时钟跳。
- **依赖物理时间戳排序事件 = bug 来源**。

### 6.2 逻辑时钟

#### Lamport Clock

每个节点维护计数器：
- 本地事件 → 计数器 +1
- 发送消息 → 带上计数器
- 收到消息 → 取 max(本地, 消息) + 1

能保证"因果序"，但**两个并发事件无法区分**。

#### Vector Clock

每个节点维护一个向量，每位是另一个节点的进度。
- 能精确判断两个事件的关系：因果序 / 并发。
- 缺点：向量大小随节点数增长。

### 6.3 Snowflake ID

分布式发号器，64 位：
```
| 1 bit | 41 bit timestamp | 10 bit machine | 12 bit seq |
```

特点：
- 趋势递增（按时间）
- 全局唯一（机器 ID + 序列）
- 单机本地生成，无外部依赖

**坑**：依赖时钟，时钟回拨会出问题。

### 6.4 TrueTime / HLC

Google Spanner 用 GPS + 原子钟提供"带误差范围的精确时间"，让跨数据中心事务可能。
HLC (Hybrid Logical Clock) 把物理 + 逻辑时钟结合，CockroachDB 等使用。

## 7. 分布式系统的常见反直觉现象

| 现象 | 含义 |
|---|---|
| **拜占庭故障 (Byzantine)** | 节点不只是死，可能发错误数据。一般工程不防（信任内网） |
| **脑裂 (Split Brain)** | 网络分区导致两边都觉得自己是 leader，写入冲突 |
| **网络分区 (Partition)** | 部分节点不通，剩余节点可能误判对方死 |
| **慢节点 (Straggler)** | 不挂但慢，比挂还麻烦（健康检查可能通过） |
| **静默丢失 (Silent Data Loss)** | 写"成功"了但实际丢了，磁盘坏块/RAID 故障 |

## 8. 反模式

| 反模式 | 痛点 |
|---|---|
| 自己造共识算法 | 99% 写错 |
| 用物理时间戳排序业务事件 | 时钟跳就出 bug |
| 重试无幂等保障 | 重复数据 |
| 分布式事务包一切 | 性能崩 |
| 副本数 = 2 | 没有多数派，不能容错 |
| 全部强一致 | 延迟和成本爆炸 |
