# 14 · 软技能：沟通、写作、决策、领导力

> 资深工程师的天花板不是技术，是**通过别人放大自己**的能力。这一章讲架构师必须的非技术能力。

---

## 1. 为什么软技能对架构师比对工程师更重要

工程师：1 个人写代码 → 1 个人的产出。
架构师：1 个决策 → 影响 10/100/1000 个工程师的产出。

**杠杆**变了。沟通失败一次，浪费一个团队一个季度。

---

## 2. 沟通

### 2.1 因人而异的沟通

| 对象 | 关心 | 怎么说 |
|---|---|---|
| **CEO / VP** | 商业影响、风险、成本、时间 | 1 张图、3 个数字、1 个决定 |
| **PM / 业务方** | 能不能做、什么时候、有什么风险 | 业务语言、场景驱动 |
| **资深工程师** | 技术细节、取舍 | 直接讲技术、欢迎挑战 |
| **新人** | 上下文、背景、为什么 | 多解释、少假设 |
| **跨团队** | 接口、依赖、影响 | 明确边界、不预设知识 |

**架构师的失败模式**：对所有人都用同一套话术。对老板讲 K8s 控制循环，对工程师讲 ROI —— 全错。

### 2.2 高密度沟通的几招

#### 2.2.1 BLUF (Bottom Line Up Front)

开场就给结论，不要"铺垫 5 分钟才说重点"。

❌ "我研究了 A、B、C 三个方案，A 是这样这样，B 是这样这样，C 是这样这样，我建议 B..."
✓ "**建议 B。** 理由：成本 1/3、性能持平、上线时间不变。下面是细节。"

#### 2.2.2 数据 > 形容词

- ❌ "性能会变好"
- ✓ "P99 从 200ms 降到 80ms"

- ❌ "成本可控"
- ✓ "月成本从 12 万降到 4 万"

#### 2.2.3 风险显性化

- 别说"基本没风险"。
- 列出来：可能失败的 3 种方式 + 概率 + 缓解。

### 2.3 听比说重要

- **复述对方的核心观点**（"你的意思是 X，对吗？"），确认理解再回。
- **找出真实需求**（"为什么"问 3-5 次到根因）。
- **沉默是工具**：对方说完别急着接，让信息沉淀。

### 2.4 跨团队对齐

最难的事。常见错误：
- 群里发完文档 = 对齐 ❌
- 开会全员点头 = 对齐 ❌

真正的对齐：
- 每个团队 owner 在文档下签名（或留 PR review approve）
- 共同 milestone 上日历
- 出问题时知道找谁

## 3. 写作

### 3.1 为什么架构师必须会写

- 文档是**异步沟通**的载体，比开会高效
- 决策需要被未来的人读懂（写 ADR）
- 个人影响力靠文档传播（你不在场时仍能改变决策）

### 3.2 好的技术文档的特征

1. **结论先行**：第一段就讲是什么、为什么、谁要看。
2. **可扫读**：标题/小节/列表，不要大段连续散文。
3. **示例驱动**：抽象概念配代码 / 图 / 真实场景。
4. **量化的承诺**：能用数字就别用形容词。
5. **承认不知道**：列出"未解决的问题"比假装全知道好。

### 3.3 文档类型与模板

#### 3.3.1 ADR (Architecture Decision Record)

每个重要技术决策一份。模板：

```markdown
# ADR-XXX: 决策标题

## 状态
[Proposed / Accepted / Deprecated / Superseded by ADR-YYY]

## 背景
- 我们现在的处境
- 我们要解决什么问题
- 约束是什么（时间、人、技术）

## 决策
我们决定：...

## 备选方案
- 方案 A：... 优点... 缺点... 为何没选
- 方案 B：...
- 方案 C：...

## 后果
- 好处：...
- 代价：...
- 风险与缓解：...

## 验证
怎么知道这个决策是对的？(指标 / 时间窗 / 复盘机制)
```

**ADR 的关键不是"我们决定了什么"，而是"我们当时为什么这样想"**。半年后你就忘了，但读到这份 ADR 能立刻回到当时的语境。

#### 3.3.2 RFC (Request for Comments)

提出方案，邀请讨论。比 ADR 早一步、未定案。

#### 3.3.3 技术方案 (Tech Spec)

包含：
- **背景与目标**
- **范围与非范围**（明确不做什么很重要）
- **方案**（高层架构 + 关键流程 + 关键数据模型）
- **接口设计**
- **可观测性与运维**
- **风险与回滚**
- **里程碑**

#### 3.3.4 Runbook

运维手册：某告警触发了怎么办？步骤化。

```
## 告警: order-service-p99-high

### 影响
订单创建延迟，可能丢失订单

### 检查
1. 看 Grafana dashboard X (链接)
2. 看 trace Y (链接)
3. 看依赖 Z 是否慢

### 缓解
- 步骤 1: 检查 DB QPS
- 步骤 2: 如果 DB 高，临时扩容只读副本
- 步骤 3: ...

### 升级
30 分钟未恢复 → 通知 SRE Lead
```

### 3.4 不该写的文档

- **逐字记录会议**：没人读
- **完整复制代码到文档**：代码会变，文档马上过期
- **超过 20 页的"完整方案"**：没人读完
- **"AI 写的没人审"的文档**：尤其严重

## 4. 决策

### 4.1 决策的分类（来自 Amazon）

| 类型 | 描述 | 怎么做 |
|---|---|---|
| **Type 1 单向门** | 不可逆（选数据库、API 设计） | 慎重、多视角、PRD 级评审 |
| **Type 2 双向门** | 可逆（功能开关、参数调整） | 快做、小步迭代、试错 |

**架构师最大的错**：把 Type 2 当 Type 1 来开会 3 个月，把 Type 1 当 Type 2 拍脑袋上。

### 4.2 RACI 矩阵（谁干什么）

| 角色 | 含义 |
|---|---|
| **R**esponsible | 谁来做 |
| **A**ccountable | 谁最终负责（**只能一人**） |
| **C**onsulted | 谁要被咨询 |
| **I**nformed | 谁要被告知 |

跨团队项目第一件事：把 RACI 写明。否则集体推卸或集体掺和。

### 4.3 决策框架：DACI / RAPID / 简单版

简单版可以是：
- **谁拍板**（Owner）
- **谁有否决权**（Veto，理想 0 人，最多 1 人）
- **谁要被咨询**
- **截止时间**

留这几个字段在文档里，决策效率立刻上去。

### 4.4 反对的艺术

> "**Disagree and Commit.**" —— Amazon 原则。

讨论时充分反对，决定后全力执行。不要"沉默赞同"再事后抱怨。

## 5. 影响力 (Leading Without Authority)

架构师常常没有人事权，要靠**影响力**让事情发生。

### 5.1 信任银行

每次准时交付、每次预测准确、每次帮人解了卡点 → 信任 +1。
每次过度承诺、每次方案翻车、每次甩锅 → 信任 -10（信任是不对称的）。

信任足够时，你说"这个要做"大家信；信任不足时，写 100 页文档也没人理。

### 5.2 怎么积累信任

- **小事可靠**：答应的 deadline 兑现，不行就提前说
- **公开成功 + 公开失败**：失败更要复盘公开
- **帮人不计较**：别人遇坑你来解，别人记一笔
- **言行一致**：你定的规范自己第一个遵守

### 5.3 让事情发生的几招

| 招 | 怎么用 |
|---|---|
| **借势** | 跟公司战略/老板关注绑定 |
| **找联盟** | 一个人推不动，找 3 个有动力的合伙人 |
| **示范** | 自己先做出来给大家看（参考实现） |
| **降低门槛** | 不光说"应该这样"，把模板/脚本/培训都准备好 |
| **量化结果** | 推完展示数字（缩短了 50% 的 X、节省了 30% 的 Y） |

## 6. 时间管理

架构师一天被会议切碎是常态。

### 6.1 反碎片化的做法

- **Focus Block**：每周固定 1-2 个半天块只写代码 / 深度文档，不让排会
- **会议批量化**：把所有会议挤到 2-3 个时段
- **拒绝旁观者会议**：你只是听的会议，问 owner 能不能事后看纪要
- **5 分钟规则**：会议 5 分钟还没找到结论，要么改 deadline 推后讨论，要么强制做决定

### 6.2 优先级管理

四象限不必教。架构师特别要警惕：
- **重要不紧急** 的事最容易被挤掉（技术债、可观测性、文档），但**它们就是架构师该做的事**
- **紧急不重要** 的事最容易吸走时间（线上小问题、各种小答疑），要么 delegate 要么挡

## 7. 培养下一代

架构师不培养接班人就走不出去。

### 7.1 怎么培养

- **写下来**：你的判断方式、踩过的坑，让别人能学
- **让出舞台**：把关键设计让给资深工程师主导，你做 reviewer
- **公开 review**：让团队一起看你的代码 review，模式可习得
- **给安全的失败空间**：让人犯能恢复的错，不要全帮兜底

### 7.2 反例

- 所有事都自己做（团队学不到）
- 只给指令不解释（培养执行者不是判断者）
- 关键评审永远自己做最后决定（团队没有 ownership）

---

## 8. 反模式

| 反模式 | 痛点 |
|---|---|
| 只对老板说"能"，不对老板说"代价" | 一开始过度承诺、后面救火 |
| 文档堆 30 页但没结论 | 没人读 |
| 决策黑箱（口头定 + 不写 ADR） | 半年后扯皮 |
| 群发文档 ≠ 对齐 | 上线时炸 |
| 自己干完所有关键代码 | 团队无成长、自己变瓶颈 |
| 沉默赞同 + 事后抱怨 | 决策无法挑战、文化恶化 |
| 用职级压人，不用逻辑说服 | 短期管用、长期失信 |
| 不分场景"对一切人讲技术细节" | 业务 / 老板听不进 |
