# 05 · 架构模式

> "模式"是被反复验证的结构模板。不必盲从，但要知道每种模式**解决什么问题**、**有什么代价**。这一章把单体、分层、六边形、Clean、微服务、模块化单体、DDD 一次讲清。

---

## 1. 单体架构 (Monolith)

### 1.1 是什么

所有功能打包成一个进程、一个部署单元。

```
┌────────────────────────────┐
│        Web / API           │
├────────────────────────────┤
│        Service Layer       │
├────────────────────────────┤
│        Data Access         │
├────────────────────────────┤
│            DB              │
└────────────────────────────┘
```

### 1.2 优点

- **简单**：一份代码、一次部署、一个调试器。
- **低延迟**：调用都是函数调用，无网络开销。
- **事务简单**：单库 ACID。
- **重构容易**：跨模块改一刀切到位，IDE 全程帮你。

### 1.3 缺点

- 团队大了之后**互相踩**（合并冲突、上线排队）。
- 任意模块崩 → 整个进程崩。
- 全栈技术栈绑死，想换语言不现实。
- 扩缩容粒度只能整个应用。

### 1.4 适用场景

- 团队 ≤ 10 人
- 业务尚未稳定，边界还在动
- MVP / 早期产品

> **"先单体后微服务"** 是工程界公认的安全路径。Martin Fowler 都说过：先把单体写好，再拆。

## 2. 分层架构 (Layered)

### 2.1 经典三层

```
Presentation  (Controller / Web)
    ↓
Business      (Service)
    ↓
Persistence   (Repository / DAO)
    ↓
DB
```

变体：N 层（加 DTO 层、Mapper 层、Facade 层）。

### 2.2 优点

- 直觉、好教、好上手
- 团队分工清晰（前端 / 后端 / DBA）

### 2.3 缺点

- **领域逻辑被淹没**：Service 层堆成"过程代码"，业务规则散落各处。
- 上层 → 下层**单向依赖**很快会被破坏（Repository 反过来调 Service 的情况）。
- 容易变成"贫血模型"：实体只有 getter/setter，业务逻辑都在 Service。

### 2.4 何时仍然合适

- CRUD 为主的系统（后台管理、内容管理）
- 团队不熟悉 DDD/Clean，不要硬上

## 3. 六边形架构 (Hexagonal / Ports & Adapters)

### 3.1 核心思想

> **业务核心**（Domain）放正中间，对外通过**端口 (Port)** 抽象，外部世界通过**适配器 (Adapter)** 接入。

```
            ┌──────────────────────────┐
            │   Adapter: REST API      │
            ▼                          ▼
       ┌────────┐                ┌────────┐
       │ Port:  │                │ Port:  │
       │ Inbound│                │Outbound│
       └───┬────┘                └───▲────┘
           ↓                         │
       ┌──────────────────────────┐  │
       │     Domain Core          │  │
       │   (Pure Business Logic)  │  │
       └──────────────────────────┘  │
                ↑                    │
       ┌────────┐                ┌────────┐
       │ Port:  │                │ Port:  │
       │  ...   │                │  DB    │
       └────────┘                └───┬────┘
                                     ▼
                              ┌──────────────┐
                              │ Adapter:     │
                              │ Postgres Impl│
                              └──────────────┘
```

### 3.2 关键约束

- Domain 不依赖任何外部技术（不 import 框架、不 import DB 驱动）。
- 外部要调 Domain，必须通过 Inbound Port。
- Domain 要用外部能力，定义 Outbound Port，让 Adapter 实现。

### 3.3 价值

- **可测**：换适配器就能跑单测，DB 用内存实现、HTTP 用直接函数调用。
- **可换**：从 MySQL 换 PostgreSQL，只换 Adapter。
- **业务核心干净**：领域代码不被框架/DB 污染。

### 3.4 代价

- 抽象多、文件多，**小项目过度工程**。
- 团队需要纪律，否则 Port 形同虚设。

## 4. Clean Architecture

### 4.1 是六边形的具体化

Robert C. Martin 提出，本质同六边形 + 明确分四层：

```
        ┌────────────────────────────────────┐
        │  Frameworks & Drivers (Web/DB/UI)  │   ← 最外层，会变
        ├────────────────────────────────────┤
        │  Interface Adapters (Controllers,  │
        │  Presenters, Gateways)             │
        ├────────────────────────────────────┤
        │  Use Cases / Application Business  │
        ├────────────────────────────────────┤
        │  Entities / Enterprise Business    │   ← 最里层，稳定
        └────────────────────────────────────┘
```

**依赖方向**：永远从外向内，里面不知道外面。

### 4.2 何时用

- 长生命周期产品（5+ 年）
- 业务复杂、规则多变
- 团队成熟，懂 DDD

不要在简单 CRUD 上硬套，会被骂"过度设计"。

## 5. 微服务 (Microservices)

### 5.1 核心特征

- 服务按业务能力拆，独立部署。
- 每个服务有自己的数据存储。
- 服务间通过网络通信（HTTP/gRPC/MQ）。
- 团队按服务自治（"two-pizza team"）。

### 5.2 拆分原则

| 错的拆法 | 对的拆法 |
|---|---|
| 按技术拆（User-Service / DB-Service / Cache-Service） | 按业务能力拆（订单 / 商品 / 库存） |
| 一拆十几个 | 先 3-5 个粗粒度，按需细拆 |
| 共享数据库 | 各服务独占数据库 |
| 拆完仍同步链长调用 | 用事件解耦，减少阻塞链 |

### 5.3 微服务税

| 代价 | 表现 |
|---|---|
| 网络开销 | 函数调用 → RPC，延迟从 ns 到 ms |
| 分布式事务 | 见 [04](04-distributed-systems.md) |
| 运维复杂 | 5 个服务 5 套 CI/CD/监控/部署 |
| 调试难 | 一个请求穿 5 个服务，要靠链路追踪 |
| 数据一致性 | 必须异步、补偿、最终一致 |
| 团队负担 | 每个团队要懂部署、监控、运维 |

### 5.4 适合上微服务的信号

- 团队 > 30 人，多个团队需要并行迭代
- 不同模块有显著不同的扩缩容/可用性要求
- 业务边界已经稳定
- 有成熟的 DevOps / 可观测性基础设施

**信号都不满足就上微服务 = 把单体的复杂度变成分布式的复杂度，且加倍。**

## 6. 模块化单体 (Modular Monolith)

### 6.1 是什么

单体的部署 + 微服务的边界。

- 一个进程、一个仓库、一次部署
- 内部强制模块边界：模块 A 不能直接 import 模块 B 的内部类
- 用 package 级 visibility / module system（Java 9 modules, .NET assemblies, Maven multi-module）强制

### 6.2 为什么这是 90% 团队的更优解

- 拿到了大部分微服务的**架构清晰度**好处
- 没有微服务的**运维 / 网络 / 一致性**代价
- **如果未来真要拆**，模块化单体已经把边界画好了，拆出去成本低

### 6.3 实现要点

```
src/
  user/
    api/         ← 对外暴露的接口（其他模块只能 import 这里）
    internal/    ← 模块内部，外部不可见
    domain/
    persistence/
  order/
    api/
    internal/
    ...
  shared-kernel/ ← 跨模块共享的稳定概念
```

代码风格上：
- 模块之间通过 **API 接口或事件总线**通信，不允许直接读对方的 DB 表。
- 跨模块共享只能放 shared-kernel，且严格审查。

## 7. 领域驱动设计 (DDD)

### 7.1 战略 vs 战术

#### 战略 DDD（更重要）

- **限界上下文 (Bounded Context)**：一个明确边界内，模型有一致的含义。
- **通用语言 (Ubiquitous Language)**：业务和开发用同一套词，"订单"在订单上下文意味着 X，在物流上下文可能意味着 Y，要明确。
- **上下文映射 (Context Map)**：上下文之间的关系（共享内核 / 客户-供应商 / 防腐层 / 开放主机服务）。

#### 战术 DDD

| 概念 | 含义 |
|---|---|
| **实体 (Entity)** | 有唯一标识、生命周期的对象（User, Order） |
| **值对象 (Value Object)** | 无标识、不可变（Money, Address） |
| **聚合 (Aggregate)** | 一组共生共死的实体，由聚合根管理一致性边界 |
| **聚合根 (Aggregate Root)** | 聚合的入口，外部只能引用根 |
| **领域服务 (Domain Service)** | 跨聚合的业务逻辑，不属于任一实体 |
| **领域事件 (Domain Event)** | "发生过的业务事实"（OrderPlaced） |
| **仓储 (Repository)** | 聚合的持久化抽象（不是 DAO） |
| **工厂 (Factory)** | 复杂聚合的构造 |

### 7.2 聚合设计的几条铁律

1. **聚合内强一致，聚合间最终一致**。
2. **一个事务只改一个聚合**。
3. **跨聚合用 ID 引用，不用直接对象引用**。
4. **聚合要小**：大聚合性能差、冲突多。

### 7.3 DDD 不适合什么

- 简单 CRUD（杀鸡用牛刀）
- 团队没有领域专家配合（变成"假 DDD"）
- 项目周期短（学习成本回不来）

DDD 更像是一种**思考方式**而非框架。学会的最大价值是"先理解业务再写代码"。

## 8. CQRS 与事件溯源

### 8.1 CQRS (Command Query Responsibility Segregation)

把**写模型**和**读模型**分开。

```
                    ┌──────────────┐
   Command (写) ──▶│ Write Model  │──▶ Event ──▶ Read Model Sync
                    └──────────────┘                    │
                                                        ▼
   Query (读)   ──────────────────────────────▶  Read Models
                                                (多个，按查询定制)
```

适合：读写比悬殊、读模式多样、写时领域复杂。

代价：最终一致、两套模型维护、复杂度高。

### 8.2 事件溯源 (Event Sourcing)

不存"当前状态"，存"所有发生过的事件"。需要当前状态时回放事件。

```
事件流：OrderCreated → ItemAdded → ItemAdded → PaymentReceived → OrderShipped
回放出：Order { items: [...], status: SHIPPED, paid: true }
```

优点：完整审计、可回到任意历史时刻、可演化新视图。
代价：模型迁移难（事件 schema 升级）、查询慢（需投影/快照）。

适合：金融账务、强审计、状态机复杂的领域。

## 9. 何时选什么 —— 决策树

```
团队 < 10 人，业务边界不清？
├─ 是 → 单体（必要时加分层）
└─ 否 → ↓

业务边界稳定、有清晰子域？
├─ 否 → 模块化单体
└─ 是 → ↓

团队 > 30 人，需要独立部署/独立扩缩容？
├─ 否 → 模块化单体
└─ 是 → ↓

有 DevOps / 可观测性 / 容器化基础？
├─ 否 → 先补基础，别上微服务
└─ 是 → 微服务

业务逻辑复杂、规则多变、长生命周期？
└─ 在所选风格内引入 DDD / Clean / 六边形
```

## 10. 反模式

| 反模式 | 表现 |
|---|---|
| **分布式单体** | 拆成了多服务，但任何改动都要同时改多个服务才能上线 |
| **共享数据库微服务** | 拆了服务但共用一个 DB schema，最坏的组合 |
| **过早微服务化** | 团队 5 人拆 15 个服务，运维瘫痪 |
| **贫血模型** | 实体只有字段，业务在 Service 里堆，DDD 形同虚设 |
| **大聚合** | 一个聚合管几十个子实体，事务冲突频发 |
| **CQRS 强加** | 简单 CRUD 上 CQRS，复杂度爆炸 |
