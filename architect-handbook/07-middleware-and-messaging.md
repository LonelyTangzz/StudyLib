# 07 · 中间件与消息

> 中间件是"业务以外的复用"。架构师不必能从零写一个 Kafka，但要能讲清它的**保证**、**代价**、**适用边界**。

---

## 1. 消息队列 (Message Queue)

### 1.1 为什么需要 MQ

| 目的 | 怎么做到 |
|---|---|
| **解耦** | 生产者不关心消费者是谁、有几个 |
| **削峰填谷** | 流量突发先入队，消费者按自己节奏处理 |
| **异步化** | 调用方不等下游处理完，返回快 |
| **广播 / Fan-out** | 一条消息多个下游各消费一份 |
| **流式处理** | 持续数据流（Kafka Streams、Flink） |

### 1.2 几款主流 MQ

| MQ | 模型 | 强项 | 弱项 |
|---|---|---|---|
| **Kafka** | 顺序日志、消费位点 | 超高吞吐、持久化、流处理 | 延迟非极致、运维重 |
| **RabbitMQ** | AMQP、复杂路由 | 灵活路由、低延迟 | 吞吐不如 Kafka |
| **RocketMQ** | 类 Kafka + 事务消息 | 国产生态、阿里系业务支持 | 国际生态弱 |
| **Pulsar** | 计算存储分离 | 多租户、跨地域、原生分层存储 | 较新、运维资料少 |
| **Redis Stream** | 简化版日志 | 部署简单（已用 Redis） | 容量小、可靠性次于 Kafka |
| **NATS / NATS JetStream** | 轻量 | 微服务间通信、低延迟 | 不适合海量持久化 |
| **AWS SQS / Azure Service Bus / GCP Pub/Sub** | 云托管 | 免运维 | 锁定、贵 |

### 1.3 投递语义

| 语义 | 含义 | 实现难度 |
|---|---|---|
| **At-most-once** | 最多一次，可能丢 | 易 |
| **At-least-once** | 至少一次，可能重 | 中（最常用） |
| **Exactly-once** | 精确一次 | 难 |

**实际工程的诚实选择是 "at-least-once + 消费幂等"**。所谓"exactly-once"（如 Kafka EOS）只在特定上下游闭环成立，跨外部副作用（发邮件、调外部 API）做不到。

### 1.4 顺序性

- **全局有序**：吞吐被锁死（单分区）。除非极端必要，不要。
- **分区内有序**：Kafka 的默认保证，按 key 路由到同一分区，同 key 有序。
- **业务顺序**：用版本号 / 时间戳在消费端处理，不依赖 MQ。

**架构含义**：把"必须有序"的范围缩小到最小键（订单内有序，订单之间不必）。

### 1.5 死信队列 (DLQ)

消费失败 N 次后，把消息搬到 DLQ：
- 不阻塞正常流
- 留住现场供后续处理
- 配告警

**DLQ 一定要有人盯**，否则等于没设。

### 1.6 消费模式

| 模式 | 描述 | 例 |
|---|---|---|
| **队列模式 (Queue)** | 一条消息只一个消费者处理 | RabbitMQ Worker、SQS |
| **发布订阅 (Pub/Sub)** | 一条消息每个订阅者各收一份 | Kafka Consumer Group、Redis Pub/Sub |
| **请求-响应 over MQ** | 模拟同步调用 | RabbitMQ RPC |

Kafka 的 Consumer Group 把两者结合：同组内是队列模式，跨组是 Pub/Sub。

## 2. 缓存 (Cache)

### 2.1 多级缓存

```
浏览器缓存
    ↓
CDN（静态资源 + 部分动态）
    ↓
网关缓存（Nginx、API Gateway）
    ↓
应用本地缓存（Caffeine、Guava Cache）   ← 微秒级
    ↓
分布式缓存（Redis、Memcached）          ← 毫秒级
    ↓
数据库
```

### 2.2 三种更新策略

| 策略 | 流程 | 优缺点 |
|---|---|---|
| **Cache Aside (旁路)** | 读：先查缓存，miss 查 DB 回填；写：写 DB + 删缓存 | 最常用、简单；短窗口可能脏 |
| **Write Through** | 写：同步写缓存 + DB | 一致性好，写慢 |
| **Write Behind** | 写：只写缓存，异步刷 DB | 写快，可能丢数据 |

**Cache Aside 是工业默认**。"写 DB → 删缓存"而非"写 DB → 更新缓存"，因为更新缓存在并发下可能写脏。

### 2.3 三大经典问题

#### 2.3.1 缓存穿透

**问题**：查询一个根本不存在的 key，每次都打到 DB。

**解决**：
- 缓存 null 值（短 TTL）
- 布隆过滤器拦在前面

#### 2.3.2 缓存击穿

**问题**：某个**热点 key** 过期瞬间，大量请求同时打到 DB。

**解决**：
- 互斥锁（同时只让一个线程去回源，别的等结果）
- 热点永不过期 + 后台异步更新

#### 2.3.3 缓存雪崩

**问题**：**大量 key 同时过期**，请求洪水打到 DB。

**解决**：
- 过期时间加随机抖动（不让一片同时过期）
- 多级缓存（一级挂还有二级）
- 限流降级

### 2.4 热点 key

单个 key QPS 极高（明星离婚、爆款商品），会打挂单台 Redis：

- 检测：Redis 4.0+ 的热 key 检测；客户端做采样统计。
- 缓解：本地缓存兜一层；key 加随机后缀做分片（读时合并）。

### 2.5 大 key

单个 value 太大（>1MB），操作慢、阻塞其他请求：

- 检测：`redis-cli --bigkeys`
- 拆分：用 Hash 拆字段、List 分页

### 2.6 缓存与 DB 一致性

**事实**：在没有分布式事务的前提下，**缓存和 DB 不可能强一致**。

工程上能做到的是"在可接受窗口内最终一致"：
- 写 DB + 删缓存（短窗口可能脏，但最终一致）
- 延迟双删（写完 DB 删一次，等几百 ms 再删一次，覆盖并发读回填的脏数据）
- 用 CDC 监听 binlog 失效缓存（解耦、可靠）

## 3. API 网关 (API Gateway)

### 3.1 是什么

所有外部流量的**统一入口**，承担：
- 路由（按 path/host 转后端）
- 认证 / 鉴权
- 限流 / 熔断
- 协议转换（HTTP → gRPC）
- 日志 / 监控
- 灰度 / AB
- 聚合（BFF 模式）

### 3.2 常见网关

| 网关 | 特点 |
|---|---|
| **Nginx / OpenResty** | 高性能、Lua 可编程 |
| **Kong** | 插件丰富、Postgres/Cassandra 后端 |
| **Apache APISIX** | 高性能、动态配置 |
| **Spring Cloud Gateway** | Java 生态、响应式 |
| **Envoy** | C++ 写、服务网格的核心 |
| **Traefik** | 自动服务发现、容器友好 |
| **AWS API Gateway / Cloudflare** | 云托管 |

### 3.3 网关 vs 反向代理

- 反向代理：四/七层转发，关注协议。
- 网关：业务级（认证、限流、版本路由），关注业务。

实际产品里两者经常合体。

### 3.4 BFF (Backend For Frontend)

为不同前端（Web、iOS、Android、小程序）各做一个轻量后端层，聚合后端服务、裁剪字段、转换协议。

价值：前端不再受"全量字段、统一格式"的束缚，性能体验好。
代价：多一层维护。

## 4. 负载均衡 (Load Balancer)

### 4.1 四层 vs 七层

| 层 | 决策依据 | 代表 |
|---|---|---|
| **四层 (L4)** | IP + 端口 | LVS、HAProxy（L4 模式）、云 NLB |
| **七层 (L7)** | HTTP 头、URL、Cookie | Nginx、Envoy、云 ALB |

四层快但不懂业务，七层灵活但有开销。一般**外层 L4 + 内层 L7** 组合。

### 4.2 负载均衡算法

| 算法 | 何时用 |
|---|---|
| 轮询 (Round Robin) | 实例同质 |
| 加权轮询 | 实例规格不同 |
| 最少连接 | 长连接服务 |
| **一致性哈希** | 需要会话粘性 / 缓存亲和性 |
| 随机 | 简单、效果接近轮询 |

### 4.3 健康检查

- **被动**：转发失败就标异常。
- **主动**：定期探活（HTTP / TCP / 自定义）。

主动 + 被动结合最稳。注意探活间隔 vs 故障感知延迟的取舍。

## 5. 服务注册与发现

### 5.1 模式

- **客户端发现**：客户端查注册中心拿可用实例，自己负载均衡。
  代表：Spring Cloud + Eureka/Nacos、gRPC + xDS。
- **服务端发现**：通过 LB / 网关转发，客户端只知道 VIP。
  代表：K8s Service、AWS ELB。

### 5.2 注册中心

- **Consul**：服务注册 + 健康检查 + KV 配置
- **Nacos**：阿里出，注册 + 配置
- **Eureka**：Netflix（已不太维护）
- **etcd / ZooKeeper**：强一致 KV，常做注册底座
- **K8s 自带**：在 K8s 环境下，CoreDNS + Service + EndpointSlice 就是注册发现

### 5.3 CP vs AP 注册中心

- **CP（ZK、etcd）**：强一致，分区时可能不可用。
- **AP（Eureka、Nacos AP 模式）**：高可用，可能短暂返回旧实例列表。

服务发现**通常选 AP**：暂时返回老实例好过整体不可用。

## 6. 服务网格 (Service Mesh)

### 6.1 是什么

把网络通信能力（重试、超时、熔断、TLS、可观测性、灰度）下沉到**与业务无关的 Sidecar 代理**，应用代码无感知。

```
┌──────────┐         ┌──────────┐
│  App A   │──────▶ │  App B   │
└──┬───────┘         └──────┬───┘
   │ localhost            ▲ localhost
   ▼                      │
┌──────────┐         ┌──────────┐
│ Sidecar  │─────────│ Sidecar  │
│ (Envoy)  │   mTLS  │ (Envoy)  │
└──────────┘         └──────────┘
       ▲                  ▲
       │  Control Plane   │
       │   (Istio etc.)   │
       └──────────────────┘
```

### 6.2 代表

- **Istio**（功能全、学习曲线陡）
- **Linkerd**（轻量、Rust 编写）
- **Consul Connect**
- **云原生**：AWS App Mesh、阿里 MSE

### 6.3 该上吗

适合：
- 多语言微服务（Java + Go + Python 都要重试/限流）
- 强治理需求（mTLS、流量管控、灰度）
- K8s 基础完备

不适合：
- 服务数少（< 20 个）
- 团队对 K8s 不熟
- 已经用 Spring Cloud / Dubbo 而且工作良好

**警告**：服务网格是新的复杂度源。不要因为"潮"而上。

## 7. 分布式锁

### 7.1 实现方式

| 方式 | 优点 | 缺点 |
|---|---|---|
| **Redis SET NX EX** | 简单、快 | 客户端 GC 等导致锁失效后还在执行 |
| **Redis RedLock** | 多节点防单点 | 算法仍有争议 |
| **ZooKeeper / etcd** | 强一致、自动释放 | 慢、运维重 |
| **DB 行锁** | 简单 | 性能差 |

### 7.2 必须注意

- **设过期时间**（防持锁进程崩了死锁）
- **锁要带 token**（防 A 释放了 B 的锁）
- **业务尽量幂等**（即使锁失效也不出错）
- **锁粒度尽量小**（按业务键，而非全局）

### 7.3 反例

"用分布式锁防重"是常见误用。**幂等是更可靠的"防重"**，锁只是辅助。

## 8. 配置中心

### 8.1 为什么

- 多环境（dev/staging/prod）配置统一管理
- 不重启动态生效
- 灰度变更
- 权限审计

### 8.2 代表

- **Nacos / Apollo / etcd / Consul**（开源）
- **AWS Parameter Store / Secrets Manager**（云）

### 8.3 关键能力

- 配置版本与回滚
- 推送（不需要客户端轮询）
- 加密敏感配置
- 灰度推送（按 IP / 标签）

## 9. 反模式

| 反模式 | 痛点 |
|---|---|
| MQ 当 DB 用（永久存储） | Kafka 是日志不是 KV，全表扫不可能 |
| 同步等 MQ 消费完返回 | 失去解耦价值 |
| 缓存当主存储 | 挂了数据全丢 |
| 全部走全局有序队列 | 并发被锁死 |
| 无 DLQ 监控 | 失败消息悄无声息 |
| 服务网格强上 | 引入新复杂度大于收益 |
| 分布式锁防一切 | 业务不幂等迟早炸 |
