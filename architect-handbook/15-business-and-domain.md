# 15 · 业务理解与领域建模

> **"不懂业务的架构师，只是高级工具人。"** 技术决策最终都为业务服务，看不到业务的人会做出"技术上完美但商业上灾难"的方案。

---

## 1. 业务理解的三个层次

| 层次 | 你能回答 |
|---|---|
| **表层**：业务流程 | 用户怎么用、订单怎么流、钱怎么算 |
| **中层**：业务模型 | 关键概念、状态、规则、约束 |
| **深层**：业务本质** | 商业模式、利润来源、护城河、增长瓶颈 |

工程师常停在表层，资深工程师懂中层，**架构师必须懂到深层**。

### 1.1 为什么深层重要

例：电商的"购物车"功能。
- 表层看：加商品、改数量、下单。
- 深层看：购物车是**留存指标**（看用户的购买意愿和回访），且承载**营销策略**（满减、优惠券应用）。

理解到深层，你设计时就不会把购物车做成"单纯的临时存储"，而会考虑：
- 购物车数据要打点到分析系统
- 营销规则需要可热更新
- 购物车 abandoned 后能否触发召回

## 2. 怎么快速理解一个陌生业务

### 2.1 一周 onboard 一个新业务的套路

**第 1 天 - 业务全景**
- 找一份"业务介绍 PPT"或"对外宣传材料"
- 跟一个业务专家聊 30 分钟：商业模式、用户、收入来源
- 列出**关键名词**（不到 50 个，多了就是没抽象）

**第 2 天 - 核心流程**
- 画 3-5 张关键流程图（下单、支付、配送、退款、对账）
- 跟客服 / 运营聊：常见问题是什么、用户为啥投诉

**第 3-4 天 - 数据**
- 看核心表 schema：能从 schema 反推业务模型
- 跑几个 SQL 看真实数据样态（用户分布、订单分布、金额分布）
- 跟 BI 聊：业务指标怎么定义、看什么报表

**第 5 天 - 钱**
- 业务怎么挣钱？毛利在哪？
- 当前的增长瓶颈在哪？技术能解什么？

**第 6 天 - 历史**
- 跟最资深的工程师聊 1 小时：哪几次架构升级、为什么、踩了什么坑
- 看几个最重要的 ADR / 复盘

**第 7 天 - 输出**
- 写一份"我对这个业务的理解" 文档
- 找业务和资深工程师 review，纠错

## 3. 领域驱动设计 (DDD) 战略层 —— 业务视角

详细概念见 [05](05-architecture-patterns.md)。这里专讲**业务和领域识别**。

### 3.1 子域 (Subdomain) 分类

| 类型 | 含义 | 投入策略 |
|---|---|---|
| **核心域 (Core Domain)** | 公司差异化的核心 | **重投入**，自研，最强工程师 |
| **支撑域 (Supporting)** | 配合核心，但通用 | 适度投入，可买可建 |
| **通用域 (Generic)** | 任何公司都有（认证、监控、消息） | **能买就买**，少自研 |

**架构师识别核心域**是关键能力。识别错了，资源就花在错的地方。

例：
- 电商：商品/订单/支付/搜索/推荐是核心；用户认证、运营后台、消息中心是支撑/通用
- 外卖：配送调度是核心；商品/订单也是核心；用户认证、消息是通用
- SaaS：业务功能是核心；多租户管理、计费是支撑

### 3.2 限界上下文 (Bounded Context) 识别

一个限界上下文 ≈ 一个团队能管的语义边界。

**识别信号**：
- **同名异义**："订单"在电商上下文是用户下的单，在物流上下文是要送的货 → 两个上下文
- **同义异名**：用户中心叫"会员"，营销系统叫"用户" → 检查是不是同一个概念
- **变更频率不同**：两块东西每次都一起改/从不一起改 → 边界信号
- **团队边界**：跨团队协作多 = 边界不清

### 3.3 通用语言 (Ubiquitous Language)

**业务和开发用同一套词**。听起来简单，做起来难。

实践：
- 关键概念建立**词典**（团队 wiki 维护）
- 代码里的类名、方法名、字段名**就用业务词**，不翻译、不造词
- 文档/邮件/PRD 里也用同一套词

**反例**：业务说"履约"，开发叫"order delivery"；业务说"客单价"，BI 叫 "AOV"。开会鸡同鸭讲。

## 4. 需求分析

### 4.1 区分"需求"和"方案"

PM 说"需要一个红色按钮" —— 那是方案。**真实需求是"让用户更容易点到付款"**。

每个需求多问 "为什么"：
- "为什么要红色按钮？" —— "测试转化率"
- "为什么要测转化率？" —— "付款流失太高"
- "付款流失高的根因是什么？" —— "页面加载慢 / 步骤多 / 信任感弱"

到根因再设计方案，**经常会发现"红色按钮"根本不是最优解**。

### 4.2 INVEST 原则（用户故事）

每个用户故事应该是：
- **I**ndependent 独立
- **N**egotiable 可协商
- **V**aluable 有价值
- **E**stimable 可估算
- **S**mall 小
- **T**estable 可测

不符合的"故事"通常是需求没拆透。

### 4.3 非功能需求 (NFR) 必须主动提

PM 一般只写功能。**架构师必须主动追问 NFR**：

| NFR | 例 |
|---|---|
| 性能 | 峰值 QPS？P99 延迟？ |
| 可用性 | SLA 几个 9？容许故障时长？ |
| 数据 | 数据量级？增长率？保留多久？ |
| 安全 | 敏感数据是什么？合规要求？ |
| 用户体验 | 设备/网络环境？多端？国际化？ |
| 可扩展 | 6 个月 / 1 年后预期规模？ |
| 运维 | 谁运维？什么时段？SLA 多久响应？ |

不问就要踩。

## 5. 业务架构 vs 应用架构

不要把这两层混。

| 层 | 关注 | 产出 |
|---|---|---|
| **业务架构** | 业务能力、流程、组织、价值流 | 业务能力地图、流程图、组织对齐图 |
| **应用架构** | 系统拆分、服务、数据、技术 | 应用清单、依赖关系、技术栈 |
| **数据架构** | 数据模型、流向、治理 | 主数据、数仓模型、ETL |
| **技术架构** | 基础设施、平台 | 部署拓扑、技术选型 |

业务架构师（在大公司是独立角色）画前两层；应用架构师画后两层；但**应用架构师必须能看懂前两层**。

### 5.1 业务能力地图

把公司业务拆成**可被多个产品复用的能力块**：

```
电商业务能力地图（简化）:

┌─────────────────────────────────────────────┐
│              用户/会员                       │
├─────────────────────────────────────────────┤
│  商品   |  库存   |  价格   |  促销         │
├─────────────────────────────────────────────┤
│  交易   |  支付   |  履约   |  售后         │
├─────────────────────────────────────────────┤
│  营销   |  会员   |  推荐   |  搜索         │
├─────────────────────────────────────────────┤
│  风控   |  财务   |  报表   |  审计         │
└─────────────────────────────────────────────┘
```

每个能力块**长期由一个团队负责**，对外提供 API/事件。**这是"中台"的真正含义**——不是又造一个层，是把能力可复用化。

## 6. 业务规则的几种类型与归属

### 6.1 分类

| 类型 | 例 | 归属 |
|---|---|---|
| **不变规则** | "订单金额必须 > 0" | Domain Entity 内（强约束） |
| **策略 (Policy)** | "VIP 用户免运费" | 可配置（规则引擎 / 配置） |
| **流程 (Workflow)** | "支付成功 → 通知履约 → 通知财务" | 编排（Saga / 工作流引擎） |
| **配置** | "国内默认顺丰、海外默认 DHL" | 配置中心 |

把它们**放对位置**是架构师的核心活：
- 不变规则放代码里硬编码（不该被改）
- 策略放规则引擎或配置（业务可改）
- 流程放编排（可视、可观测、可改）

### 6.2 反例

- 把"VIP 免运费"硬编码进代码 → 业务想加"白金会员免运费"要发版
- 把"订单金额必须 > 0"放配置中心 → 配错就出灾难（金额可负）

## 7. 业务指标驱动技术

### 7.1 找业务的关键指标 (North Star Metric)

每个业务都有一个最高指标，比如：
- 电商：GMV、订单数
- 内容：DAU、停留时长
- SaaS：ARR、续约率
- 工具：留存、付费转化

技术决策要**能回答"这能怎么撬动 NSM"**。否则就是技术 ego。

### 7.2 指标的拆解

NSM → 一级指标 → 二级指标 → 三级指标 → 技术指标。

例（电商）：
```
GMV
 ├── 流量 × 转化率 × 客单价
 │    ├── 流量
 │    │   ├── 自然流量
 │    │   ├── 付费流量
 │    │   └── 社交分享 ← 技术能优化分享体验
 │    ├── 转化率
 │    │   ├── 详情页转化 ← 技术能优化加载速度
 │    │   ├── 购物车转化
 │    │   └── 支付成功率 ← 技术能优化支付稳定性
 │    └── 客单价
 │        ├── 推荐相关性 ← 技术能优化算法
 │        └── 促销策略效果
```

架构师 sitting in 业务讨论时，能在脑子里画这棵树，就能立刻判断"这个技术投入值不值"。

## 8. 与业务方协作的几个原则

### 8.1 "假装是业务方" (Dogfooding)

自己当用户用产品。一周用一次就能发现一堆问题，比看 50 份需求文档管用。

### 8.2 用业务语言，不用技术词压人

跟业务方说："我们要重构服务网格、上 Istio 实现金丝雀发布" —— 业务听不懂、不关心、不批预算。

改成："我能让线上变更出问题时 1 分钟回滚，目前要 10 分钟" —— 业务秒懂、愿意支持。

### 8.3 量化业务影响

每个技术 proposal 都要尝试回答：
- 这能给业务带来什么具体好处（指标 +/- 多少）？
- 风险是什么？最坏情况业务怎样？
- 没做会怎样？

### 8.4 不做"业务的传声筒"

PM 提需求，你照单转给开发 = 失职。
**架构师要 push back**：
- 这个需求和上次那个冲突
- 这个需求有更好的实现路径
- 这个需求 ROI 太低、性价比不够

业务方一开始可能不爽，长期会感激（避免做了无意义的活）。

## 9. 反模式

| 反模式 | 痛点 |
|---|---|
| "技术不需要懂业务" | 做出业务无价值的精美方案 |
| 把核心域当通用域（觉得太普通） | 该自研的买、被竞争对手甩开 |
| 把通用域当核心域 (NIH) | 自研一堆"认证系统"、"消息系统"、烧人力 |
| 业务术语和代码命名不一致 | 跨团队鸡同鸭讲 |
| 业务规则硬编码 | 每改一次要发版 |
| 不主动问 NFR | 上线就炸 |
| 拒绝 push back 任何需求 | 团队疲于做无价值的事 |
| 用技术术语跟业务/老板沟通 | 资源拿不到 |
